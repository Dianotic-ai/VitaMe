## 2. 五大Claude Code方法论框架深度解析

### 2.1 slavingia/skills（~7.9K星）

**是什么：** Sahil Lavingia（Gumroad创始人）把《极简创业》一书转化为**10个可执行的Claude Code斜杠命令** —— `/validate-idea`、`/mvp`、`/pricing`、`/first-customers`等。

**核心模式：** 每个skill是一个SKILL.md文件，包含YAML元数据 + 角色定义 + 3-5步流程 + 红绿灯决策标准 + 明确结论。**模式本身比具体内容更有价值** —— 任何领域知识都可以编码成这种格式。

**对你项目最有用的几个：**
- **`/validate-idea`** —— 核心问题："你能在写代码之前就把这个卖出去吗？" 对保健品领域意味着：先找10个愿意付¥19.9/月的人
- **`/processize`** —— "先手动交付服务再自动化"。你们可以先在微信上手动做保健品咨询来验证需求
- **`/mvp`** —— "周末就能上线，范围最小化" 完全匹配黑客松约束
- **`/pricing`** —— 测试¥19.9定价的结构化框架

**关键洞察：** 这些skill是**苏格拉底式提问者**，不是内容生成器。它们问你问题、挑战你的假设，而不是直接产出。这种对抗式思维比"帮我生成一个商业计划"效果好得多。

### 2.2 BMAD-METHOD（~43K星）

**是什么：** 完整的AI驱动软件开发生命周期框架，定义了**6个专家角色**：
- 分析师Mary（需求分析）
- 产品经理John（PRD创建）
- 架构师Winston（技术设计）
- 开发者Amelia（编码实现）
- UX设计师Sally（交互设计）
- 技术作家Paige（文档编写）

**4阶段工作流：** 分析 → 规划 → 方案设计 → 实现。每个阶段产出结构化文档，作为下一个角色的输入上下文。

**核心创新 —— Epic Sharding（史诗拆分）：** Scrum Master把需求拆成自包含的story文件，每个story嵌入了开发者代理需要的**全部上下文**。这解决了Claude"上下文崩塌"的核心问题。

**对黑客松最重要的：** 别用完整的4阶段流程。用 **`bmad-quick-dev`**（快速开发模式）：
1. 压缩意图（几句话就够）
2. 生成规格说明
3. 实现（用子代理）
4. 审查（无上下文子代理审查）

你们已经完成了Phase 1（分析），现在应该直接跳到快速开发模式。

### 2.3 garrytan/gstack（~72K星）

**是什么：** YC总裁Garry Tan的Claude Code配置，包含**23个角色命令**，围绕冲刺周期组织：Think → Plan → Build → Review → Test → Ship → Reflect。

**对你最有价值的命令：**

- **`/office-hours`** —— YC的6个逼问框架（最重要的命令）：
  1. **需求真实性**：你能说出一个具体的、需要这个的人吗？
  2. **现状**：他们今天怎么解决这个问题？
  3. **绝望程度**：他们是否绝望到愿意用一个不完美的v1？
  4. **最窄切入点**：你能做的最小的证明论点的东西是什么？
  5. **独特观察**：你观察到了什么别人没注意到的？
  6. **未来适配**：这个东西会随时间变得更重要吗？

- **`/cso`** —— OWASP Top 10 + STRIDE威胁建模。**处理健康数据必须用这个，不可跳过。**
- **`/autoplan`** —— 自动链接 CEO审查 → 设计审查 → 工程审查，一条命令跑完
- **`/qa`** —— 打开真正的Chromium浏览器做端到端测试
- **`/ship`** —— 最终部署流水线

Garry Tan声称用这套系统**60天写了60万行生产代码**。

### 2.4 obra/superpowers（~152K星，最受欢迎）

**是什么：** 引入了**Subagent-Driven Development (SDD) —— 子代理驱动开发**方法论。

**核心创新：**
- 每个微任务派发一个新鲜的子代理（限定在2-5分钟内完成）
- **双阶段审查**：先查规格符合性，再查代码质量
- 4种状态信号：DONE（完成）、DONE_WITH_CONCERNS（完成但有顾虑）、BLOCKED（阻塞）、FAILED（失败）
- **技能自动触发** —— Claude检测你在做什么然后自动调用相关skill，不需要手动输入斜杠命令
- **强制TDD**：如果Claude先写代码再写测试，superpowers会删除代码并强制重来

**任务分解示例：** `writing-plans` skill把工作拆解成2-5分钟任务，每个任务包含确切的文件路径、完整的代码规格说明和验证步骤 —— 设计目标是"一个热情但缺乏品味、没有判断力、不了解项目上下文的初级工程师也能执行"。**这恰好就是AI辅助新手开发者需要的粒度。**

### 2.5 everything-claude-code（~102K星）

**是什么：** 基础设施层 —— **28个预构建代理、119个技能、60个命令**，覆盖12种语言生态。

**最实用的组件：**
- **PM2 + 多代理编排**：`/multi-plan`、`/multi-execute`、`/orchestrate` 用于复杂并行工作流
- **AgentShield**：102条安全规则 + 912个测试 —— 对健康数据处理很重要
- **商业技能**（v1.7.0+）：`market-research`、`investor-materials`、`content-engine` —— 直接用于黑客松演讲准备
- **内存持久化钩子**：跨会话保存上下文
- **持续学习系统**：从已完成工作中提取模式形成可复用技能

**作者Affaan Mustafa是Anthropic黑客松冠军。**

### ⚡ 五个框架如何组合（不冲突）

它们在**三个不同抽象层**运作：

| 层级 | 框架 | 在你项目中的角色 |
|------|------|-----------------|
| 项目生命周期 & 编排 | BMAD-METHOD | PRD创建、架构文档、冲刺规划、故事拆分 |
| 创业判断 & 产品决策 | gstack + slavingia/skills | 产品验证、范围决策、安全审计、发布 |
| 工程方法论 & 执行 | superpowers + everything-claude-code | TDD、子代理驱动开发、调试、代理基础设施 |

**推荐的使用顺序：**
1. **slavingia/skills** → 第0阶段：用 `/validate-idea` 和 `/pricing` 在写代码前验证商业想法
2. **gstack `/office-hours`** → 用YC的6个逼问来磨利产品定位
3. **BMAD `bmad-quick-dev`** → 压缩 意图→规格→实现→审查 周期
4. **superpowers** → 子代理驱动实现 + 自动双阶段审查 + TDD
5. **everything-claude-code** → 代理定义、安全规则、项目结构脚手架

**潜在冲突管理：** BMAD和gstack都定义了角色——规划阶段用BMAD角色，审查/发布阶段用gstack角色。superpowers的自动触发可能与BMAD工作流状态冲突——在CLAUDE.md中配置哪个框架管理哪个阶段。

## 3. Claude Code子代理(Subagent)工作原理

### 基本机制

Claude Code的 **Agent工具**（原名Task工具，v2.1.63后改名）用于生成子代理。每个子代理获得：
- **独立的200K token上下文窗口**
- 专门的系统提示
- 可配置的工具权限（Read读取、Write写入、Edit编辑、Bash命令、Grep搜索、Glob匹配）
- 每次生成约**20K token的上下文加载开销**

**关键点：** 子代理**无法读取父级对话历史**。父级到子级的唯一通道是调用时的prompt字符串。

### 三种执行模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **前台执行** | 阻塞主对话，等待完成 | 关键路径任务 |
| **后台执行** | 并发运行（Ctrl+B），权限预批准 | 非阻塞分析/研究 |
| **并行派发** | 最多约**10个并发任务**，超出部分排队 | 独立域并行开发 |

### 协调机制

**文件系统是主要协调机制。** 代理把计划写到 `./plans/*.md`，状态更新到共享YAML文件，实现产物到各自的目录。**CLAUDE.md被每个会话和子代理自动加载**，是编排中最重要的单一文件。

你可以在不同目录层级放置CLAUDE.md —— 所有层级的CLAUDE.md都会被加入上下文，从而实现子目录级别的专门指令。

### ⚠️ 关键限制（必须知道的坑）

1. **不能嵌套委派**：子代理不能再生成子代理。句号。
2. **子代理之间完全隔离**：并行子代理看不到彼此的工作
3. **文件冲突风险**：两个子代理同时编辑同一文件会导致合并冲突 → 解决方案：按清晰的文件边界分区工作
4. **上下文窗口成本翻倍**：每个子代理消耗自己的token。Agent Teams模式的token消耗约为单会话的**7倍**
5. **命名冲突陷阱**：名为 `code-reviewer` 的子代理会被预定义的通用规则覆盖自定义指令 → 使用不明显的名称

### 实际案例

**Anthropic C编译器项目：** 使用16个代理，约2,000个会话（$20,000 API成本），产出100,000行Rust C编译器。关键教训：测试必须极其高质量，结构化状态文件帮助代理自我定位。

**YC创业公司案例：** Ambral（多子代理客户智能引擎）、Vulcan Technologies（非技术创始人用Claude Code构建合规产品）、HumanLayer（12因素代理模式）。

## 4. 商业模式硬伤分析（挑战）

### 4.1 CPS佣金率被严重高估

项目假设5-15%的CPS佣金率，但实际平台数据完全不同：

| 平台 | 保健品品类实际佣金率 | 说明 |
|------|---------------------|------|
| 京东联盟（自营） | **0.5-3%** | 2024Q3公告专门下调了医药保健品类佣金 |
| 淘宝联盟（标准） | **1.5-3%** | 8-15%需要定向计划，要求有销量基础和商家直接合作 |
| 淘宝联盟（扣费后） | 名义10% → 实际~9% | 扣除10%技术服务费 |

**对一个新的、没有流量证明的平台，现实的有效佣金率是2-5%。**

### 4.2 每用户收入（ARPU）算不过账

| 收入来源 | 现实假设 | 每MAU月收入 |
|---------|---------|------------|
| CPS佣金 | 5%用户月购买 × ¥150客单价 × 3%佣金 | **¥0.23** |
| 订阅 | 3%转化率 × ¥19.9/月 | **¥0.60** |
| **混合ARPU** | | **~¥0.83** |

以¥0.83的ARPU：
- 覆盖¥500/月运营成本 → 需要**602 MAU**
- 覆盖¥50,000/月（两人团队现实运营成本）→ 需要**60,241 MAU**

参考：Keep有2,992万MAU，ARPU ¥5.75/月 —— **仍然亏损。**

### 4.3 一颗法律地雷：《广告法》第46条

**中国《广告法》第46条** 要求所有保健食品广告必须经过前置审查。2023年《互联网广告管理办法》明确禁止 **"利用健康/养生知识内容变相发布保健食品广告"** —— 这恰好描述了一个通过AI推荐具体保健品并从中赚取佣金的应用。

处罚可达**100万元以上罚款**。这不是理论风险，是中国保健品监管的结构性特征。

### 4.4 没有成功的中国健康App用CPS作为核心收入

| 公司 | 主要收入模式 | CPS角色 |
|------|-------------|---------|
| **丁香医生** | B2B医药营销服务（8000万微信粉丝+300万认证医生网络）| 非核心 |
| **薄荷健康** | 彻底转型为DTC健康食品自有品牌（40-60%毛利）| 完全放弃了工具模式 |
| **Keep** | 自有品牌产品46% + 会员44% + 广告9%（全年营收20.66亿）| 不用CPS |

### 4.5 更好的变现路径

| 路径 | 单位经济 | 优势 |
|------|---------|------|
| **保险导流** | ¥50-200/合格线索 | 远超保健品CPS；平安健康服务1,198家企业客户 |
| **企业健康管理** | ¥50-500/员工/年 | 市场规模4,175亿元，CAGR 8.12%；可预测、高毛利 |
| **分层订阅** | ¥9.9基础 / ¥19.9标准 / ¥39.9高级 | 降低转化门槛，增加ARPU |

**建议：** 黑客松阶段不要在Demo中展示CPS佣金模式，改为口述"保险导流+企业B2B"作为商业故事。

## 5. 竞品分析再深入

### 5.1 蚂蚁阿福 —— 为什么主动不碰保健品

蚂蚁阿福（2024年12月由"AQ"更名）是中国排名第一的AI健康App，**3000万+月活、1000万+日查询**。它提供健康问答、报告解读、医生连接、健康档案 —— 但**明确排除保健品推荐**。

更有意思的是，蚂蚁阿福甚至推出了**"AI打假"功能**，帮用户识别虚假保健品广告 —— 把自己定位为保健品商业的对立面。蚂蚁健康业务负责人公开表态："我们的商业模式在内部争议很大；我们还没有答案。"

**字节跳动的小荷AI医生**（2025年中上线，基于豆包大模型）也呈现同样模式 —— 全面的健康AI但零保健品功能。两家公司都引用了监管风险、责任暴露和中国老年群体保健品骗局的声誉顾虑。

### 5.2 最接近的中国竞品是品牌锁定的

**Key科益家**（赫力昂/GSK × 欧飞基因）运营微信小程序，结合健康问卷和可选基因检测数据生成个性化保健品推荐。但它**只推荐赫力昂自有产品** —— 本质上是品牌营销工具，不是品牌中立的健康顾问。不做成分形式级分析，不做药物-保健品交互检查。

**LemonBox**（YC投资，100万+微信用户）同样提供AI驱动的个性化维生素订阅，但只卖自有DTC产品，没有交互检查。两者验证了中国消费者对AI保健品指导的需求，但都是品牌锁定的。

### 5.3 全球范围内，拼图碎片存在但没人组装

| 产品 | 能力 | 缺什么 |
|------|------|--------|
| **Persona Nutrition**（雀巢健康科学，美国）| 4,000+药物-营养素交互，1,000+处方药筛查 | 品牌锁定、仅美国 |
| **Supp.AI**（Allen AI研究所）| 59,096个补充剂-药物交互，从2200万篇论文中提取 | 仅研究工具，非消费者产品 |
| **VerifiedSupplementData.com** | 成分形式级质量分析（区分甘氨酸镁vs氧化镁）| 仅评测网站，非推荐引擎 |
| **CanIStackThis.com** | AI交互检查 | 不做个性化推荐 |
| **VitalStack** | AI补充剂交互检查（Claude & ChatGPT接入）| 不做推荐 |

**全球没有一个产品统一了：品牌中立推荐 + 成分形式分析 + 禁忌交叉校验 + 个性化，放在一个消费者工具里。**

### 5.4 缝隙真实存在，但存在结构性原因

这个缝隙存在是因为：
1. **成分形式级分析需要深厚药学专业知识** —— 科技公司没有
2. **补充剂禁忌数据库远不如药物交互数据库完善**（尤其是中国传统补品）
3. **品牌中立定位从根本上与最明显的变现路径（卖补品）冲突**

这些不是等着聪明创业者来解决的问题 —— 它们是这个领域固有的张力，解释了为什么拥有3000万用户的蚂蚁阿福选择不进入。

**风险不是没人会进入这个空间，而是进入壁垒（监管、责任、数据基础设施、信任）足够高使得2人黑客松团队可能是在不稳定地基上建造 —— 但又足够低使得如果模式被证明可行，蚂蚁或字节可以用压倒性的分发优势复制它。**

## 6. 技术架构方案重构

### 6.1 微信小程序是17天内的不可能任务

**2023年9月起，所有微信小程序都需要ICP（互联网内容提供商）备案。** 标准流程**至少3-4周**，健康相关内容需要额外审批更长。每次代码更新需要微信审核（每次1-3个工作日）。API端点必须预先白名单。保健品推荐工具几乎肯定触发健康类目资质要求，增加未知的额外等待时间。

**这一个监管事实就使得4月30日上线微信小程序成为不可能。**

建议：**立即启动ICP备案作为并行工作流，但黑客松MVP用Web App构建。**

### 6.2 推荐的17天技术栈

| 层级 | 技术 | 为什么选它 |
|------|------|-----------|
| **前端** | Next.js 14+ (App Router) 部署到 Vercel | 即时部署，不需要ICP，通过二维码在微信浏览器内打开 |
| **后端** | FastAPI (Python) 部署到 Railway/Render | 完美匹配Kevin的Python背景，自动生成API文档 |

| **禁忌数据** | SUPP.AI API + 硬编码关键规则 | 59,096条交互记录，免费，有证据支撑 |
| **开发工具** | Cursor AI + Claude Code（配superpowers框架）| 加速学习中的开发者达到交付速度 |

### 6.3 三层禁忌检查引擎

**第1层 —— SUPP.AI API**（第1-2天）：
- 免费REST API，包含2,044种补充剂、2,866种药物、59,096条交互记录
- 从2200万篇科学文章中提取的证据句子
- 38.4MB数据集可下载用于离线缓存

**第2层 —— 硬编码关键规则**（第2-3天）：
- 一个JSON/SQLite表，包含~50-100条最危险的交互（贯叶连翘+SSRIs、银杏+抗凝血药、高剂量维E+抗凝剂）
- 作为安全网独立于LLM准确性

**第3层 —— LLM验证**（第3-4天）：
- 对第1-2层未覆盖的交互，传给DeepSeek并要求结构化引用来源和显式表达不确定性
- **永远不要仅依赖LLM做安全关键的禁忌决策**

### 6.4 跳过RAG，用系统提示注入

DeepSeek V3.2有**128K上下文窗口** —— MVP规模完全不需要RAG。整理一个结构化JSON知识库（100-200种常见补充剂的功效、剂量、证据质量、禁忌），作为系统提示注入（约20-30K token）。DeepSeek的缓存命中价格仅$0.028/百万token，重复系统提示几乎免费。

后期知识库超过系统提示容量再加ChromaDB。

### 6.5 DeepSeek安全需要自建护栏

DeepSeek V3.2在成本上完胜（$0.28 vs Claude Sonnet的$3.00/百万输入token），但Cisco/宾大研究发现DeepSeek R1**对100%的越狱提示未能阻挡**。对健康AI，必须构建三层安全体系：

1. **输入过滤**（LLM前）：紧急关键词 → 重定向到"请拨打120"；检测诊断/处方请求 → 注入免责声明；阻断prompt提取攻击
2. **系统提示工程**：硬编码规则包括"始终包含医疗免责声明"、"永不诊断疾病"、"用⚠️标记所有潜在交互"、强制不确定性语言
3. **输出过滤**（LLM后）：正则扫描危险声明（"治疗""治愈""诊断"），确保每个响应包含免责声明，记录所有交互
