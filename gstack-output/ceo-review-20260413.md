## Section 1: 合规与法律风险深度分析

这是整个review最重要的一节。用户问了三个问题，逐一回答。

### Q1: 会不会面临合规和风控问题？

**答案：会。但风险是可管理的，前提是做对几件事。**

**中国法律框架：**

| 法规 | 与你的关系 | 风险等级 |
|------|-----------|---------|
| 《食品安全法》 | 保健品属于"保健食品"，推荐不等于销售，但若用户因推荐受损可能被追责 | 高 |
| 《广告法》第18条 | 保健食品广告不得含有疗效承诺、不得利用科研成果做广告 | 中 |
| 《个人信息保护法》 | 体检数据=敏感个人信息，需单独授权+加密+最小化 | 中 |
| 《反不正当竞争法》 | CPS推荐若被认定为"虚假宣传"，面临罚款 | 中 |
| 《消费者权益保护法》 | 用户因推荐购买后不满意，可能要求退一赔三 | 中 |

**真实案例警示：**
- 广州颜如玉：因在APP/微信渠道宣称普通食品能"改善骨质疏松""心血管疾病"，被罚100万元
- 中国法院对食品安全违规一般适用"退一赔十"（10倍赔偿）
- 涉及虚假专家推荐保健品的刑事追诉案例存在

**关键红线：你的产品绝对不能越过"保健≠治疗"的线。**

**法规强制声明：** 根据《广告法》第18条，所有保健食品相关内容必须显著标明：**"保健食品不是药物，不能代替药物治疗疾病"**。这不是可选的免责声明，是法定义务。

**法律定位：** 国家卫健委界定：提供饮食或生活习惯建议属于"在线咨询/健康信息服务"，不属于《互联网诊疗管理办法》管辖。但一旦涉及对疾病下诊断结论或治疗方案，就构成"诊疗行为"，必须持有《医疗机构执业许可证》。你的"生活服务-营养查询"定位是合法路径。

现有防御措施（设计文档中已有）：
- [x] 免责声明每次AI输出强制展示
- [x] 严重异常硬编码触发"请立即就医"
- [x] 禁忌规则硬编码不依赖LLM
- [x] 措辞避免"治疗""治愈""药效"

**需要补充的防御措施：**
- [ ] **法律实体+知识产权从第一天就设立** — Cydoc（运营7年的健康AI创业公司）最大的遗憾就是忽略了法律/财务基础设施
- [ ] **推荐成分，不推荐品牌** — 学习Examine.com模式：只分析证据，不推荐具体产品。CPS链接只在用户主动要求时展示
- [ ] **输出审计日志** — 每次AI推荐完整记录（输入、输出、规则触发），用于事后追溯
- [ ] **保险** — 了解"专业责任保险"（Professional Liability Insurance）是否适用
- [ ] **蓝帽子校验** — 只推荐已获得NMPA注册/备案的"蓝帽子"产品，功能声称必须与注册批准内容一致
- [ ] **法定声明** — UI固定展示"保健食品不是药物，不能代替药物治疗疾病"（《广告法》第18条强制要求）

### Q2: 为什么大厂不做？

**不是因为违法，是因为风险/收益比不划算。**

蚂蚁阿福（3000万月活）不做保健品推荐的真正原因：

| 因素 | 大厂视角 | 创业公司视角 |
|------|---------|------------|
| 收益 | 保健品CPS佣金~5-15%，对蚂蚁集团营收贡献微乎其微 | 这是主要营收来源 |
| 风险 | 一个用户投诉 = 媒体头条 = 品牌危机 | 早期用户理解产品阶段性 |
| 合规成本 | 需建专门合规团队审核所有推荐内容 | 50条规则+1个药剂师审核 |
| 战略优先级 | 有100个更赚钱的方向 | 这就是唯一方向 |
| 责任边界 | 3000万用户的任何出错都是灾难 | 100个种子用户的反馈循环 |

**一句话：大厂不做不是因为不能做，是因为不值得做。对于一个创业公司来说，这正好是机会。**

但这也意味着：**当你长到一定规模（比如100万用户），你会面临和大厂一样的问题。** 合规基础设施必须在那之前建好。

### Q3: 证据来源的可信度

**"为什么别人会相信我们的推荐？"**

这是产品生死问题。信任 = 可验证的证据链。

**证据来源分级：**

| 等级 | 来源 | 可信度 | 可用性 | 你怎么用 |
|------|------|--------|--------|---------|
| Tier 1 | NIH Office of Dietary Supplements (ODS) | 最高，美国政府机构 | 英文，免费公开 | 核心引用源 |
| Tier 1 | Examine.com | 高，独立第三方，不接受品牌赞助 | 英文，部分免费 | 证据等级评估 |
| Tier 2 | DSLD (Dietary Supplement Label Database) | 高，FDA管理 | 英文API | 成分标签验证 |
| Tier 2 | PubMed系统评价/Meta分析 | 高，同行评审 | 英文，免费 | 具体剂量建议的证据 |
| Tier 3 | 国家药监局(NMPA)保健食品数据库 | 中，覆盖国内注册产品 | 中文 | 蓝帽子标识验证 |
| Tier 3 | 中国营养学会DRIs | 中，国内权威 | 中文 | 参考摄入量 |
| ⚠️ | DeepSeek生成内容 | 低，可能幻觉 | 中文 | **绝不作为唯一证据源** |

**⚠️ 重要发现：NIH数据访问受限风险**

2025年4月起，美国NIH已禁止中国用户访问SEER等数据库，PubMed访问也面临不确定性。这意味着：
- NIH ODS Fact Sheets目前仍可访问，但未来可能受限
- **降级方案：** 在成分知识库构建阶段批量下载并本地化所有需要的NIH/PubMed数据，不依赖实时API调用
- **合规注意：** 引用外国文献不能替代中国SAMR批准的功能声称。中国保健食品只能宣传《保健食品功能目录》内的24项功能
- **建议：** 以中国官方批准的功能声称为准，外国文献仅作为补充参考

**关键设计原则：AI是翻译器，不是信息源。**

所有推荐必须可追溯到Tier 1或Tier 2来源。DeepSeek的角色是把英文文献翻译成用户能理解的中文建议，不是自己编造建议。

**信任建设策略：**

1. **每条推荐标注证据来源和等级**
   - "基于NIH ODS维生素C Fact Sheet（最后更新2024年3月）"
   - "证据等级：强（多项系统评价支持）" vs "证据有限（仅小规模研究）"

2. **不确定时明确说"不知道"**
   - 成分库未收录 → "我们的数据库暂未收录此成分，建议咨询药剂师"
   - 证据不足 → "目前研究证据有限，无法给出明确建议"

3. **可验证性**
   - 用户可以点击引用链接直接看到原文（英文，但有中文摘要）
   - 这比"相信我"强100倍

4. **Examine.com模式的智慧**
   - Examine.com从不推荐品牌，从不接受广告，从不与品牌合作
   - 这是为什么它被全球营养师信任的原因
   - 你的MVP应该学这个：**推荐成分+剂量+化学形式，不推荐具体品牌**
   - CPS链接只在用户主动搜索购买时出现（"我想买"按钮），不夹在推荐里

## Section 2: 假阴性问题 — 50条规则的致命缺口

这是Premise 4的深层问题，也是整个项目最大的技术风险。

### 问题定义

你计划硬编码~50条禁忌规则。但现实中：
- NIH ODS记录的药物-补剂交互有**数百种**
- Natural Medicines Database记录了**上千种**补剂-补剂交互
- 每年新增数十项研究发现新的交互关系

50条 ≈ 覆盖最常见的5-10%场景。

**危险场景：** 用户输入一个不在你50条规则里的组合，系统返回"绿色-无冲突"。用户以为安全，实际上存在交互风险。

**这比没有冲突检测功能更危险。** 因为用户建立了"这个工具说安全就是安全"的认知。

### 解决方案：分级覆盖 + 诚实边界

```
检测结果不是二元的"安全/不安全"，而是三种：

🔴 已知冲突 → "检测到冲突：[具体说明]"
🟢 已知安全 → "在我们的数据库中，这些成分无已知冲突"
⚪ 未覆盖   → "这个组合不在我们的检测范围内，建议咨询药剂师"
```

**关键改变：** 添加第三种状态"⚪ 未覆盖"。绝不对不在数据库中的组合说"安全"。

**实施方式：**
- 50条红色规则（硬编码禁忌）
- 50条绿色规则（硬编码已知安全组合）
- 其他所有组合 → 灰色"未覆盖"状态
- UI上明确展示覆盖率："本次检测覆盖了您输入成分的X/Y种可能组合"

**MVP工作量增加：** 需要额外整理50条"已知安全"组合（约多2-3小时），但这解决了整个项目最大的法律风险。

### 禁忌规则数据来源可行性

| 数据源 | 交互数据量 | 可用性 | MVP适用 |
|--------|-----------|--------|---------|
| NIH ODS Fact Sheets | ~30种主要维生素/矿物质的交互 | 免费，英文 | 是 |
| Examine.com交互页面 | ~200种补剂的交互关系 | 部分免费 | 是 |
| Natural Medicines Database | 1000+种 | 付费，约$100/年 | 考虑Growth阶段 |
| PubMed Case Reports | 持续增长 | 免费 | 选择性引用 |

**MVP建议：** 用NIH ODS + Examine.com免费数据整理Top 50禁忌 + Top 50安全组合。其余组合标记为"未覆盖"。

### 关键数据流 + 边缘情况

**Flow 1: 冲突检测（Tab2核心功能）**
```
用户输入保健品名 → 名称标准化 → 成分提取 → 交叉校验禁忌库
                                                ↓
                              ┌─────────────────┼─────────────────┐
                              ↓                 ↓                 ↓
                          🔴 已知冲突       ⚪ 未覆盖         🟢 已知安全
                          展示警告+原因     展示"超出范围"     展示安全确认
                          +替代方案         +建议咨询药剂师    +证据来源
```

**边缘情况：**
- 用户输入品牌名而非成分名（如"汤臣倍健钙片"而非"碳酸钙"）→ 需要品牌→成分的映射表
- 用户输入中英文混合（"vitamin C"+"钙片"）→ 需要双语标准化
- 同一成分不同剂量的不同结论（500mg维C安全，2000mg维C对肾结石患者危险）→ 需要剂量上下文
- 用户未录入病史就做冲突检测 → 只检测补剂间交互，提醒"录入病史获取更完整检测"

**Flow 2: AI对话推荐（Tab1核心功能）**
```
用户输入体检数据+症状 → FastAPI接收
  → Step 1: 硬编码异常检测（血糖>7.0等）→ 命中则强制返回"请就医"
  → Step 2: 构建DeepSeek Prompt（用户数据+成分知识库检索结果+角色约束）
  → Step 3: DeepSeek API调用
  → Step 4: 输出过滤（禁忌校验+关键词过滤+免责声明注入）
  → 返回前端展示
```

**边缘情况：**
- DeepSeek API超时（>15s）→ 返回预缓存的通用建议 + "AI暂时繁忙"提示
- DeepSeek输出包含禁忌成分推荐 → 输出过滤层拦截，替换为安全推荐
- DeepSeek编造引用（如"根据2024年Lancet研究..."实际不存在）→ **这是最难防的边缘情况**
- 用户在对话中透露紧急情况（"我胸痛"）→ 需要关键词触发强制就医

**DeepSeek幻觉引用问题的解决方案：**
- MVP阶段：Prompt约束"只引用以下来源列表中的内容"，把成分知识库的引用信息直接注入Context
- 如果DeepSeek引用了Context之外的来源 → 输出过滤层标记为"[来源待验证]"
- Growth阶段：用RAG替代直接Prompt注入，引用链接可点击跳转原文

|--------|------|------|--------|
| 单次深度报告 | ¥9.9 | 用户愿意为AI解读付费吗？蚂蚁阿福免费做体检解读 | ⚠️ 低 — 需要证明比免费方案好很多 |
| CPS佣金 | 5-15% | 如果学Examine.com模式不推荐品牌，CPS从哪来？ | ⚠️ 需要重新设计 — "我想买"按钮而非主动推荐 |
| 订阅 | ¥19.9/月 | 健康App订阅留存极低，4% 30天留存率 | ⚠️ 高风险 — 需要证明持续价值 |

**核心问题：** 如果遵循Examine.com的"不推荐品牌"模式来降低法律风险，CPS收入模型就需要调整。

**建议的CPS模式调整：**
```
当前设计：AI推荐 → 推荐结果中嵌入CPS链接
问题：推荐和佣金混在一起，即使声称"解耦"，用户和监管都会质疑

修改建议：
1. AI推荐 → 只输出成分名+化学形式+剂量（无品牌）
2. 用户看完推荐后，主动点击"帮我找产品"按钮
3. 进入独立的产品搜索页面（明确标注"以下为购买建议，含合作链接"）
4. 搜索结果按成分匹配度排序，不按佣金排序

好处：推荐环节100%中立，购买环节自愿进入，法律责任更清晰
```

### 竞争壁垒耐久度

**真正的护城河不是技术，是数据+信任。** 技术可以被复制，但用户3年的体检趋势数据和对平台的信任无法复制。

## Section 5: 用户体验与留存策略

### 留存问题的根源

设计文档已经识别了4%的30天留存率问题。但分析还不够深。

**健康App低留存的三个层次：**

| 层次 | 原因 | 你的应对 | 充分吗？ |
|------|------|---------|---------|
| 表层 | 初始流程繁琐 | 手动录入5个指标即可 | ✅ 够 |
| 中层 | 没有持续使用理由 | 冲突检测作为高频入口 | ⚠️ 频率被高估 |
| 深层 | 健康改善感知缓慢 | 纵向趋势图展示变化 | ❌ 需要至少2次体检数据 |

**被忽略的留存钩子：** "成分知识库浏览"

用户买保健品前搜一下"这个成分怎么样"是一个真实的使用场景，频率可能比冲突检测更高。Tab2（成分安全中心）应该包含一个独立的"成分百科"入口，类似Examine.com的中文简化版。

**建议增加的功能点（MVP可选，低开发成本）：**
- "今日成分知识"推送（每天一条小知识，利用微信服务通知）
- 成分搜索入口（用户搜"镁"→ 看到3种化学形式对比表）
- 这两个都可以用现有成分知识库JSON驱动，不需要额外数据

### Demo UX审查

设计文档的5分钟Demo结构是好的，但有一个问题：

**问题：** Demo从Tab2（冲突检测）开始，然后切到Tab1（AI对话）。评委的心理路径是"冲突检测 → AI对话 → 商业模式"。但产品的日常使用路径是"AI对话 → 冲突检测 → 购买"。Demo顺序和使用顺序不一致，评委可能会问"用户日常怎么用？"

**建议：** Demo保持恐惧开场（冲突检测），但在第3分钟快速演示"一个真实用户的日常使用流程"（30秒）：打开小程序 → 看到上次体检趋势 → 点"帮我推荐" → 看到建议 → 点"帮我找产品"。这样评委同时看到了"杀手功能"和"日常使用场景"。

## Section 7: Outside Voice（独立第二意见）

**模拟一个对健康科技持怀疑态度的投资人的视角：**

"你的产品核心假设是'用户在乎保健品成分的化学形式差异'。但大多数买保健品的人连成分表都不看——他们看的是品牌、价格、小红书博主推荐。

你的真正用户不是9亿亚健康人群，是那些已经在主动研究保健品成分的人。这个群体有多大？可能是总人群的1-3%。

好消息是：这1-3%是最有价值的用户。他们愿意付费，愿意为了健康花时间。坏消息是：你的获客成本会很高，因为你需要找到这些人。

冲突检测是个聪明的裂变点——'你吃的这几样冲突吗？'这个问题对所有吃保健品的人都有吸引力，不只是研究成分的人。这是你的增长引擎。

但最大的问号是：你怎么从'工具'变成'平台'？如果用户只在买新保健品时来用一次，你就是一个低频工具。你需要给用户一个'每周回来'的理由，目前的计划里这个还不够清晰。"

**Outside Voice的建议：**
1. MVP只做冲突检测+AI对话，这是对的
2. 不要急着做CPS变现，先证明留存
3. 找5个真实用户（不是朋友），给他们用产品，看他们一周后还会不会回来
4. 如果不回来，产品需要pivot，不管黑客松结果怎样

## VERDICT

**STATUS: DONE_WITH_CONCERNS**

**Overall Assessment: 7.5/10**

设计方向正确，双Tab方案在17天限制下是最优选择。核心差异化（成分形式级推荐+禁忌安全引擎）真实存在且有市场空白。

**3个必须在编码前解决的问题：**

1. **假阴性问题**（最重要）：添加三级分类系统（红/绿/灰），绝不对未覆盖的组合说"安全"。这不是nice-to-have，是法律生死线。

2. **CPS与推荐分离**：推荐环节只输出成分（不含品牌），"帮我找产品"作为独立入口。这降低了"推荐≈广告"的合规风险。

3. **证据链可追溯**：每条推荐必须标注来源（NIH/Examine/PubMed），DeepSeek不能作为唯一信息源。Prompt设计必须约束AI只引用已知来源。

**做对了的地方：**
- 选择创业赛道（产品+商业+技术三维评审匹配）
- 恐惧开场Demo策略（冲突检测制造moment）
- 禁忌规则硬编码不依赖LLM
- ¥74/月的成本结构
- 腾讯AI小程序计划的资源利用

**下一步：** 解决上述3个问题后，可以开始编码。建议先跑 `/plan-eng-review` 做技术架构的工程级审查，然后进入开发。
