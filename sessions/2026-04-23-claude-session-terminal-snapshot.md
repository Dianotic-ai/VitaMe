  四、核心设计①：Verify-first 多入口

  原则：5 秒内必须见到价值。

  - 产品定位"25-40 岁成年子女给父母选购"——主场景就是"拿到一瓶东西想知道能不能吃"，所以 ④Verify 必须能独立先跑
  - 入口不是"请填写你的健康信息"，而是"粘个链接 / 拍个瓶子 / 问一句"
  - Tier 0 的输出永远标注："基于通用人群估计——告诉我 X 可让判断个性化 ✅/❌"，用钩子牵引升档

  原则：5 秒内必须见到价值。

  - 产品定位"25-40
  岁成年子女给父母选购"——主场景就是"拿到一瓶东西想知道能不能吃"，所以 ④Verify
  必须能独立先跑
  - 入口不是"请填写你的健康信息"，而是"粘个链接 / 拍个瓶子 / 问一句"
  - Tier 0 的输出永远标注："基于通用人群估计——告诉我 X 可让判断个性化
  ✅/❌"，用钩子牵引升档

  十、v1 → v2 的关键改动

  1. 入口从 ①Intake 前置改为 ④Verify 可先跑（解决冷启动流失）
  2. ①Intake 从"一次 5 分钟"改为"四档渐进、按需升档"（解决新用户怕填）
  3. 隐私从"后端实现"升格为"UI 常驻 HUD"（解决隐私顾虑）
  4. Memory 从"默认建档"改为"用户明示同意才建"（信任边界）
  5. 每轮 LLM 输出加"钩子句式"：告诉我 X 可让这条变准 → 把升档做成产品体验而非额外步骤

  ---
  ★ Insight ─────────────────────────────────────
  - "产品即建档"是冷启动的唯一解：Gemini 是"先问病史再给建议"，VitaMe 应该是"先给建议再顺手收集一项信息"。每一次交互都在建档，但用户感知是在"拿到价值"——这是所有成功消费型
   AI 产品（Cal AI、Finch、Flo）的共同模式。
  - 隐私的产品化关键不是"更安全"，是"更可见"：同样是本地加密，用户看得到开关 vs 藏在 FAQ 里，信任感差一个数量级。结合健康类产品的特殊性（用户天然警惕），把隐私 HUD 做成主
   UI 元素，反而会变成"别人抄不走的差异点"。
  - Tier 0 能跑"通用人群模式"的前提：50 条硬编码禁忌规则必须能在"没有具体档案"时也触发（按商品成分 × 常见高风险人群交叉打标）。这反过来定义了规则库 v0
  的最低完备度——不只是给个性化用，也是 Tier 0 的引擎。
