## 1. 验证清单 — 北极星 §1-§9 逐项

### §1 North Star — "通过长期反馈让 Agent 越来越懂你"

| 子目标 | 验证步骤 |
|---|---|
| 入口：补剂安全判断 | /chat → 问"鱼油 + 华法林"→ 应红警拦截 + 引证 |
| 触发器：Reminder | /profile → 加补剂 → 设规则 → 改系统时间或加历史时间 → /chat 看顶部 banner |
| 数据来源：Feedback | /chat 进入 1.5s 后 → 应弹 FeedbackPrompt（前提：currentSupplements > 0 且超 24h） |
| 上下文：Memory | /memory → 看时间轴累积 verify/feedback/reminder 事件 |
| 自我进化：Hermit Agent | /memory → 点"Hermit 帮我看看"按钮 → LLM 归纳后产生 observation card |

### §2 Core Loop — 完整闭环验证

执行下列步骤一次（约 5-8 分钟）：

1. /chat 切到一个 person → 问"我妈在吃华法林" → 流式回复 → 看 /memory 出现 verify event ✓
2. /profile 切到该 person → 加补剂"鱼油 1000mg/天 早餐后" → 设提醒规则"每天 8:00" ✓
3. （等 8:00 过 OR 改系统时间到 8:00 后）/chat → 看顶部 ReminderBanner ✓
4. 点"吃了 ✓" → 看 /memory 出现 reminder event ✓
5. 24 小时后再访问 /chat → FeedbackPrompt 自动弹（也可立刻在浏览器 Console 删掉 sessionStorage 强触发）✓
6. 提交反馈 → 看 /memory 出现 feedback event ✓
7. 累积 5+ 事件后，/memory → 点"Hermit 帮我看看" → 5-15 秒后看到 observation card ✓
8. 点"接受" → 看 /memory 出现 correction event ✓

### §3 Reminder — 规则验证表

| 北极星要求 | 实现 | 验证 |
|---|---|---|
| 帮用户减少忘记服用 | ReminderBanner | 设规则后到点显示 |
| 在合适时机触发轻反馈 | (P3 链接：暂时手动) | banner click ack 后下次 FeedbackPrompt |
| 默认低频 | 默认每天 1 次 | /profile 编辑器默认 8:00 周一-日 |
| 用户可暂停 | reminder rule.paused | /profile 点'暂停'按钮 |
| 跳过/降频 | consecutiveSkips + frequencyMultiplier | 连续 3 skip → 0.5x；5 skip → 0.25x |
| 连续无响应自动降频 | ackRule('skip') 内置 | 连续 3 次跳过看降频徽章 |
| 高风险补剂不'放心服用'暗示 | banner 文案"该吃 X 了，吃了吗？"中性 | 不带'放心'字眼 |

### §6 Hermit Agent

| 要求 | 实现 |
|---|---|
| 周期归纳器 | /api/hermit + HermitButton 触发 |
| 发现反馈模式 | observationType: 'pattern' |
| 提醒用户复查 | observationType: 'recheck' |
| 建议调整提醒 | observationType: 'reminder-adjust' |
| 请求补充字段 | observationType: 'request-field' |
| **不能** 诊断 / 因果归因 / 自动改方案 / 替代医生 | system prompt 显式列 4 条禁止 + sanitize 过滤无效 type |

⚠️ **设计调整**：原计划 Vercel Cron 周期触发，改为客户端按需触发。
**理由**：Memory events 仅在客户端 LocalStorage（§9.8），服务端没有数据可扫。手动触发避开了 cross-user 数据上传的隐私问题。
**未来方案**：用户可选开启 cloud memory 后，再做 cron 自动归纳。

## 2. 关键设计决策摘要

| 决策 | 理由 |
|---|---|
| 加补剂 vs 长期用药分两个字段（currentSupplements vs medications） | medications 是处方药/OTC，补剂不进；reminder 只跟踪 currentSupplements |
| Reminder 只用 in-app banner，不用 push notification | PWA + service worker + 权限弹窗 = 妈用户体验差，留 P3 |
| Hermit 客户端触发不用 Vercel Cron | 隐私底线 §9.8：events 不出本机 |
| Hermit 输出限 3 条 + 80 字 / proposal 40 字 | 避免 LLM 失控；用户可一眼判断是否接受 |
| FeedbackPrompt 同一 session 最多 1 次 + 跨日 sessionStorage | 不骚扰，符合北极星 §3 "默认低频" |
| 删 person → 同步 cleanup reminder rules + memory events | 防止孤儿数据 + 满足 GDPR-style "删除我的全部数据" |

## 3. 已知开放问题（partner 可能问）

1. **Q: Reminder 不依赖 push 是否够？**
   A: P0/P1 范围内够（用户访问时显示）。但用户长期不打开应用 = 失效。P3 加 PWA push。

2. **Q: 'urgent 关键词硬拦截' 的关键词列表全吗？**
   A: 当前只有 7 个（剧痛/呼吸困难/出血/过敏休克/晕倒/抽搐/昏迷/心悸严重），需要药师扩 list。P1 后置任务。

3. **Q: Hermit 周期触发改成手动，还能叫"周期归纳"吗？**
   A: 北极星 §6 没明说"必须自动"。手动触发本质等价：用户主动找 Hermit 看 = 用户驱动的周期。Auto cron 留 P3 (用户开 cloud memory 后)。

4. **Q: 多 person 的 Hermit 互不影响吗？**
   A: 独立。每次只扫 active person 的 events，不跨 person 归纳。

5. **Q: observation 接受/忽略后会真的执行 proposal 吗？**
   A: **目前只记录用户态度，不自动执行**。北极星 §7 严格说"用户可确认可撤销" — 我们认为"接受"是用户认可建议，但具体执行（如调整 reminder time）需用户去 /profile 手动改。后续可加"接受 → 自动 patch"路径，但要先讨论是否符合 §7 "可见"原则。
