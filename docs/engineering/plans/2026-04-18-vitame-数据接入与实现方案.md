## 1. 系统架构总览

### 1.1 三层 Mental Model(对标常见 Web 分层)

**专业术语**:Three-tier data architecture(呈现层 / 逻辑层 / 适配层)。
**大白话**:**字典 + 报警器 + 翻译器**——字典告诉你这成分是啥,报警器告诉你能不能吃,翻译器把瓶子上的英文字变成结构化输入。

```
┌─────────────────────────────────────────────────────────────────┐
│                      用户 H5 (微信 WebView)                      │
│             输入:文字 / 拍照瓶子 / 选择已有档案                    │
└────────────────────────────────┬────────────────────────────────┘
                                 │
                                 ▼
┌─────────────────────────────────────────────────────────────────┐
│            L3 适配层 (Query Intake + OCR)                        │
│  ─────────────────────────────────────────────────────────       │
│  职责:无论输入是什么,输出统一的 [成分+剂量+形式] 列表            │
│                                                                 │
│  子模块:                                                         │
│   • InputNormalizer    中文同义词 → 标准英文成分名               │
│   • OCRReader          瓶子照片 → 成分文本(Minimax 多模态)      │
│   • ProductMatcher     产品名 → DSLD/TGA/機能性/蓝帽子 SKU 查询 │
│   • IntakeOrchestrator 串联 + 兜底逻辑                           │
│                                                                 │
│  数据源:DSLD Top 500 + TGA Swisse + 機能性 DHC + 蓝帽子 Top 30 │
└────────────────────────────────┬────────────────────────────────┘
                                 │ 标准化 [{ingredient, dose, form}]
                                 ▼
┌─────────────────────────────────────────────────────────────────┐
│              L2 判断层 (Safety Judgment)                         │
│  ─────────────────────────────────────────────────────────       │
│  职责:给定成分列表 + 用户病史/在服药 → 红黄灰绿判定              │
│                                                                 │
│  子模块:                                                         │
│   • RuleRegistry       注册 3 种 rule source                    │
│   • HardcodedLookup    50 条硬编码禁忌(病史×成分)              │
│   • SuppaiLookup       补剂×药物交互查询                         │
│   • DdinterLookup      药×药交互查询(P1)                       │
│   • RiskLevelMerger    多路结果合并 + 取最严                     │
│                                                                 │
│  数据源:contraindications.ts + suppai-rules.ts + ddinter.ts    │
└────────────────────────────────┬────────────────────────────────┘
                                 │ Risk[] (带 source_ref)
                                 ▼
┌─────────────────────────────────────────────────────────────────┐
│              L1 呈现层 (Safety Translation)                      │
│  ─────────────────────────────────────────────────────────       │
│  职责:Risk 对象 → 用户看得懂的中文(含吸收率对比、替代建议)       │
│                                                                 │
│  子模块:                                                         │
│   • PromptBuilder      组装 LLM Prompt(含成分知识 + Risk)       │
│   • LlmAdapter         Minimax / DeepSeek / OpenClaw 三路切换   │
│   • TemplateFallback   LLM 失败时的兜底模板                     │
│   • FormComparator     成分形式对比(氧化镁 vs 甘氨酸镁)         │
│   • GuardrailFilter    合规过滤(禁词 + 医疗措辞)                 │
│                                                                 │
│  数据源:ingredients.ts(NIH ODS + LPI + 中国 DRIs)              │
└────────────────────────────────┬────────────────────────────────┘
                                 │ 中文结果 + 证据链
                                 ▼
            [Compliance middleware: Evidence/Disclaimer/Audit]
                                 ▼
                       前端渲染结果页 RiskCard
```

### 1.2 和 p0-plan.md 的映射关系

| p0-plan.md 里的模块 | 本文档的层 | 补强内容 |
|---|---|---|
| `src/lib/query/*` | L3 | 新增 OCRReader 子模块 + 多国 ProductMatcher |
| `src/lib/judgment/*` | L2 | 已对齐,无变更 |
| `src/lib/translation/*` | L1 | 成分知识来源从 30 条扩到 50 条 |
| `src/lib/adapters/*` | 接入辅助 | 加 tga.ts / kinoseihyouji.ts / bluehat.ts |
| `src/lib/db/*` | 所有层的数据 | 从 2 个 db 文件扩到 7 个 |
| `scripts/bake*.ts` | 烘焙脚本 | 从 2 个扩到 7 个 |

### 1.3 本架构的设计原则(给 CC 看的)

1. **运行时不联网**:所有知识库烘焙成 `src/lib/db/*.ts`,build 进 bundle
2. **source_ref 强制**:任何输出到用户的 Risk 对象必须带 `source_ref`,用于合规追溯
3. **LLM 不查事实**:LLM 只负责"翻译 + 组织语言",事实(禁忌/剂量/交互)从 db 来
4. **三路合并取最严**:L2 三个 lookup 并行跑,取最高风险等级
5. **无 key 不跑**:任何涉及外部 API 的代码,若环境变量未设置则 skip + warn,不中断构建

## 2. 数据接入层详细设计

### 2.1 数据流水线总览

每个数据源走同一条流水线,分 3 步:

```
[下载原始数据] → [bake 脚本转换] → [src/lib/db/*.ts] → [adapter 查询封装]
      ↓                ↓                    ↓                    ↓
  一次性(人工)    一次性(npm run bake)   构建时产物          运行时调用
```

**大白话对比**:
- 传统 Web:**数据库(MySQL) → ORM(Prisma) → Service**
- VitaMe P0:**TS 常量(db/*.ts) → Adapter 函数(adapters/*.ts) → Service**
- 本质是一样的,只是**把数据库查询简化成内存 Map 查找**,因为数据量小(总 <5MB)且不变。

### 2.3 核心 TypeScript 契约(CC 开发必须遵守)

所有 db/ 文件遵循这套 interface,这样 adapter 层可以统一处理。

```typescript
// src/types/ingredient.ts
export interface Ingredient {
  id: string;                          // 'magnesium'
  nameEn: string;                      // 'Magnesium'
  nameZh: string;                      // '镁'
  aliases: string[];                   // ['Mg', '镁元素', '天然镁']
  forms: IngredientForm[];             // 镁有 4 种形式
  dri: DRIReference;                   // 中美两套剂量参考
  contraindications: string[];         // 关联的 contraindication id
  sourceRefs: SourceRef[];             // 证据来源
}

export interface IngredientForm {
  nameEn: string;                      // 'Magnesium Glycinate'
  nameZh: string;                      // '甘氨酸镁'
  absorptionRate?: number;             // 0.3 (30%)
  notes: string;                       // '助眠,刺激小,推荐敏感人群'
}

export interface DRIReference {
  us: { rdi?: number; ul?: number; unit: string };  // 美国 NIH
  cn: { rni?: number; ul?: number; unit: string };  // 中国营养学会
}

export interface SourceRef {
  source: 'NIH_ODS' | 'LPI' | 'CNS_DRI' | 'SUPP_AI' | 'DDINTER' | 'DSLD' | 'TGA' | 'JP_KINOSEI' | 'CN_BLUEHAT' | 'VITAME_HARDCODED';
  url?: string;
  retrievedAt: string;                 // ISO date
  note?: string;
}

// src/types/interaction.ts
export interface Interaction {
  id: string;                          // 'suppai-warfarin-fish-oil'
  substanceA: { id: string; type: 'supplement' | 'drug' };
  substanceB: { id: string; type: 'supplement' | 'drug' };
  severity: 'red' | 'yellow' | 'gray' | 'green';
  reason: string;                      // '增加出血风险'
  doseThreshold?: string;              // '>3g/天'
  sourceRef: SourceRef;
}

// src/types/product.ts
export interface Product {
  sku: string;                         // DSLD id / TGA AUST L / 蓝帽子批号
  brand: string;
  nameOriginal: string;
  nameZh?: string;
  country: 'US' | 'AU' | 'JP' | 'CN';
  ingredients: { id: string; dose: number; unit: string; form?: string }[];
  upc?: string;                        // 条形码(美国有,其他不一定)
  sourceRef: SourceRef;
}
```

### 2.4 每个数据源的烘焙脚本规范

#### 2.4.1 NIH ODS Fact Sheets → `ingredients.ts`(L1 主力)

**原始数据**:30 篇 HTML,URL 形如 `https://ods.od.nih.gov/factsheets/{nutrient}-HealthProfessional/`

**烘焙流程**:
```
1. scripts/raw/ 手工下载 30 篇 HTML(或用 fetch 脚本)
2. scripts/bakeNihFactSheets.ts:
   - HTML → 纯文本(cheerio)
   - 按固定 section 切片(Background / Recommended Intakes / Sources / Intakes / Deficiency / Health Effects / Interactions)
   - 关键字段提取:RDI / UL / 推荐剂量
   - 调 Minimax 批量翻译 + 结构化为 JSON
   - 输出到 src/lib/db/ingredients.ts(覆盖式写入)
3. 人工审核(必须):翻译准确性 + 数值一致性
```

**产物体积**:30 成分 × ~5KB = **~150KB**
**工程量**:1 人日(LLM 跑 2h + 人工审 6h)
**命令**:`npm run bake:nih`

#### 2.4.2 Linus Pauling Institute → 合并入 `ingredients.ts`

**原始数据**:40 篇 HTML,`https://lpi.oregonstate.edu/mic/{nutrient}`
**和 NIH 的差异**:LPI 更偏"健康影响"和"疾病关联",补 NIH 的"循证"部分
**烘焙流程**:同上,但产物是对 ingredients.ts 里已有条目的 `healthEffects` / `evidenceLevel` 字段 **增量更新**
**工程量**:0.5 人日
**命令**:`npm run bake:lpi`

#### 2.4.3 中国营养学会 DRIs → 合并入 `ingredients.ts`

**原始数据**:《中国居民膳食营养素参考摄入量(2023 版)》公开节选 PDF / 网络资料
**烘焙流程**:
```
1. 手工录入 50 成分的 CN RNI / CN UL 值到 JSON
2. scripts/bakeCnDri.ts 合并进 ingredients.ts 的 dri.cn 字段
```
**关键**:**不翻译全书**,只取关键数值,合理使用范围
**工程量**:0.5 人日
**命令**:`npm run bake:cn-dri`

#### 2.4.4 DSLD → `dsld-ingredients.ts`(L3 字典,**不是产品库**)

**这里和 p0-plan.md 的 T-0.22 有重大变更**:

| 维度 | p0-plan.md 原设计 | 本文档修正设计 | 理由 |
|---|---|---|---|
| 烘焙对象 | Top 3000 产品 + 成分映射 | **Top 500 成分名字典** | 产品匹配由 OCR + LLM 兜底,不需 63 万 SKU |
| 产物体积 | ~10MB | ~50KB | 减 200× |
| 产物文件 | dsld-products.ts | **dsld-ingredients.ts** | 文件改名 |
| 用途 | ProductMatcher | InputNormalizer(英文成分标准化) | 职能变了 |

**新的烘焙流程**:
```
1. 下载 DSLD JSON dump(官方提供的 dump 文件)
2. scripts/bakeDsld.ts:
   - 扫描所有产品的成分字段
   - 统计成分出现频次
   - 取 Top 500 成分(预估覆盖 95% iHerb 主流产品)
   - 产出 { standardName, aliases, chebiId?, pubchemCid? } 字典
3. 产物 = 英文成分标准化词典
```

**为什么这么改**:OCR + LLM 读图后,输出的已经是"magnesium glycinate 200mg",这时需要的是**标准化词典**(把 "Magnesium Bisglycinate" "Chelated Magnesium" "Mg Glycinate" 都规约到 magnesium.glycinate),而不是全量产品 SKU 库。

**工程量**:0.5 人日
**命令**:`npm run bake:dsld`

#### 2.4.5 SUPP.AI → `suppai-interactions.ts`

**原始数据**:CSV,5.9 万条
**烘焙流程**:
```
1. 下载 SUPP.AI CSV(https://supp.ai/data)
2. scripts/bakeSuppai.ts:
   - 过滤:涉及 Top 50 成分(ingredients.ts 的 id 白名单)
   - 过滤:涉及 Top 100 常见药物(手工维护药物白名单)
   - 预估剩余 ~1500 条
   - 分类 severity(CSV 里有 interaction_type 字段)
3. 产出 Interaction[]
```
**工程量**:1 人日(其中白名单维护 0.3 天)
**命令**:`npm run bake:suppai`

#### 2.4.6 TGA ARTG → `tga-products.ts`

**原始数据**:澳大利亚 Therapeutic Goods Administration 公开数据库
**目标品牌**:Swisse / Blackmores / Bioisland / Bio Island / Ostelin
**烘焙流程**:
```
1. https://www.tga.gov.au/resources/artg — 支持按 sponsor 搜索
2. scripts/bakeTga.ts:
   - 对 5 个目标品牌各抽 Top 40 产品
   - 提取:AUST L 编号 + 产品名 + 成分列表
   - 预估 ~200 条
```
**工程量**:0.3 人日
**命令**:`npm run bake:tga`

#### 2.4.7 日本機能性表示食品 → `japan-products.ts`

**原始数据**:消费者庁公开 Excel(https://www.caa.go.jp/policies/policy/food_labeling/foods_with_function_claims/)
**目标品牌**:DHC / FANCL / Orihiro / Kobayashi
**烘焙流程**:
```
1. 下载全量 Excel(数万行)
2. scripts/bakeKinoseihyouji.ts:
   - 按届出者(ブランド)过滤
   - 预估剩 ~150 条
   - 提取:届出番号 + 商品名 + 機能性関与成分
```
**工程量**:0.3 人日
**命令**:`npm run bake:jp`

#### 2.4.8 蓝帽子 Top 30 → `china-products.ts`

**原始数据**:http://ypzsx.gsxt.gov.cn/specialfood/ (国家市场监督管理总局)
**烘焙流程**:
```
1. 手工(或极简爬虫)拿 30 个主流国产品的注册信息
   - 汤臣倍健(蛋白粉/多种维生素/钙/褪黑素等 15 个 SKU)
   - 善存(2 个)
   - 钙尔奇(3 个)
   - 斯利安(2 个)
   - 养生堂(3 个天然维生素系列)
   - 其他(5 个)
2. scripts/bakeBluehat.ts 合并成 JSON
```
**工程量**:0.5 人日(手录为主)
**命令**:`npm run bake:bluehat`

#### 2.4.9 DDInter → `ddinter-drug-drug.ts`(P1 加分)

**原始数据**:http://ddinter.scbdd.com/download/
**烘焙流程**:类似 SUPP.AI,筛选 Top 100 药物子集,~500 条
**工程量**:0.5 人日
**命令**:`npm run bake:ddinter`

#### 2.4.10 手工 `contraindications.ts`(50 条硬编码)

**这个不是烘焙,是人工编写 + 朋友审核**

```
1. 基于 product-brief 蒸馏包里的禁忌列表,扩展到 50 条
2. 病史类目:肾结石/肾功能不全/肝病/甲亢/甲减/孕期/服华法林/服 SSRI/服甲状腺素
3. 每条:condition + ingredient + threshold + severity + reason + source_ref
4. Demo 前必须找药剂师审核(本周起在网上找,不阻塞开发)
```
**工程量**:0.5 人日撰写 + 外部审核
**无烘焙命令**(直接写 TS 文件)

### 2.5 Adapter 层接口规范(给 CC 看)

所有 adapter 遵循同一签名,便于 Safety Judgment 并行调用:

```typescript
// src/lib/adapters/types.ts
export interface LookupRequest {
  ingredients: IngredientRef[];        // 输入的成分列表
  context: {
    conditions?: string[];             // 病史
    medications?: string[];            // 在服药
    demographics?: { age?: number; sex?: string; pregnancy?: boolean };
  };
}

export interface LookupResponse {
  risks: Risk[];                       // 命中的风险项
  sourceName: string;                  // 'SUPP.AI' / 'VITAME_HARDCODED' / ...
  latencyMs: number;                   // 用于性能监控
}

export interface Adapter {
  name: string;
  lookup(req: LookupRequest): Promise<LookupResponse>;
}

// 实例
export const suppaiAdapter: Adapter = { name: 'SUPP.AI', lookup: ... };
export const hardcodedAdapter: Adapter = { name: 'VITAME_HARDCODED', lookup: ... };
export const ddinterAdapter: Adapter = { name: 'DDInter', lookup: ... };
```

Safety Judgment 的核心代码:

```typescript
// src/lib/judgment/judgmentEngine.ts
export async function judge(req: LookupRequest): Promise<Risk[]> {
  const adapters = [hardcodedAdapter, suppaiAdapter, ddinterAdapter];
  const results = await Promise.all(adapters.map(a => a.lookup(req)));
  return mergeRisks(results.flatMap(r => r.risks));   // 取最严
}
```

## 3. 运行时数据流(端到端链路)

### 3.1 场景 A:用户拍照输入(Demo 主场景)

```
[用户] 拍照: Now Foods Magnesium Glycinate 200mg 的瓶子
    │
    ▼
[OCRReader] 调 Minimax 多模态
    输入: 图片 base64
    Prompt: "识别补剂标签,输出 JSON: {brand, product_name, ingredients:[{name,dose,unit,form}]}"
    输出: { brand: "Now Foods", product_name: "...", ingredients: [{name:"Magnesium Glycinate", dose:200, unit:"mg"}] }
    │
    ▼
[InputNormalizer] 查 dsld-ingredients.ts
    "Magnesium Glycinate" → standardId: "magnesium.glycinate"
    │
    ▼
[ProductMatcher] 可选:查 DSLD/TGA/JP/CN,获取完整产品信息
    (Demo 里用户已经拿着瓶子,这步可以简化或跳过)
    │
    ▼
[ContextCollector] 问用户 3-5 个关键问题
    "你有哪些已知疾病?""你正在吃什么药?"
    │
    ▼
[SafetyJudgment] 三路并行
    • HardcodedLookup:病史 × 成分 → 命中
    • SuppaiLookup:  成分 × 在服药 → 命中
    • DdinterLookup: 药 × 药(辅助) → 命中
    • RiskLevelMerger:取最严
    │
    ▼
[SafetyTranslation] LLM 把 Risk 翻译成用户语言
    输入: Risk 对象 + ingredients.ts 里该成分的 L1 知识
    Prompt 模板: "你是补剂安全助手...基于以下事实:{facts}...输出中文建议..."
    LLM: Minimax (fallback: DeepSeek / OpenClaw)
    输出: { zhSummary, zhReason, zhAvoidance, evidenceLinks }
    │
    ▼
[Compliance middleware]
    • EvidenceAnnotator:每条建议附 source_ref
    • BannedPhraseFilter:剔除"治疗/治愈/药效"等词
    • CriticalEscalation:极高风险强制加"请立即就医"
    • DisclaimerInjector:免责声明
    • AuditLogger:写 JSONL
    │
    ▼
[前端 RiskCard] 渲染结果:红/黄/灰/绿 + 原因 + 替代建议 + 来源按钮
```

### 3.2 场景 B:用户文字输入(备选)

同上,但跳过 OCRReader,InputNormalizer 直接处理中文输入("我想吃鱼油和维D" → ["omega-3", "vitamin.d"])。

### 3.3 场景 C:已有档案复查(Demo 亮点)

```
[用户点"妈妈"档案] → [新增钙片] → [复查所有成分组合]
    │
    ▼
[RecheckOrchestrator] 把档案里已有成分 + 新成分 一起传给 SafetyJudgment
    (关键:不重复问既有病史,直接用档案里的 context)
```

### 3.4 性能目标

| 场景 | 目标 | 关键瓶颈 |
|---|---|---|
| OCR 识别 | < 5 秒 | Minimax 多模态延迟 |
| 完整 query → result | < 10 秒 | LLM 翻译 |
| 纯 SafetyJudgment(不带翻译) | < 500 ms | 内存 Map 查找 |
| 复查已有档案 | < 3 秒 | 无 OCR,只走 L2+L1 |

## 4. LLM 层使用策略

### 4.1 三路 Adapter 职责分工

| Provider | 角色 | 场景 |
|---|---|---|
| **Minimax**(默认) | 主力 | OCR 多模态 + 结果翻译(文本) |
| **DeepSeek**(备选) | Fallback | Minimax 配额爆 / 延迟高 |
| **OpenClaw Gateway**(本地) | 最终兜底 | 网络抖动 / 评委现场网差 |

切换靠 `process.env.LLM_PROVIDER`,CC 直接用 OpenClaw 官方源码,无需自研。

### 4.2 Prompt 设计原则

1. **事实从 DB 来,LLM 只组织语言**:Prompt 里塞 ingredients.ts 的原始 JSON,让 LLM 基于此输出,不允许 LLM 自己编剂量/禁忌
2. **结构化输出**:Prompt 强制 `response_format: { type: 'json_object' }`,用 Zod schema 校验
3. **温度 0.2**:避免创造性发挥
4. **Token 预算**:单次 query 总 token < 8K(输入 5K + 输出 3K),Minimax 能扛

### 4.3 OCR Prompt 模板(关键)

```
你是专业的膳食补充剂标签识别专家。请从图片中提取以下信息,严格按 JSON 输出:

{
  "brand": "品牌名(英文)",
  "product_name": "产品名(英文)",
  "country_guess": "US|AU|JP|CN|UNKNOWN",
  "ingredients": [
    {
      "name_en": "成分英文名(用 Supplement Facts 原文)",
      "dose_value": 数字,
      "dose_unit": "mg|mcg|IU|g",
      "form_hint": "glycinate/oxide/etc",
      "daily_value_percent": 数字或 null
    }
  ],
  "confidence": "high|medium|low",
  "unreadable_parts": ["无法识别的部分描述"]
}

要求:
- 只提取 Supplement Facts / 成分表 部分,忽略营销语
- 剂量必须从标签上读,不要猜
- 无法识别时 confidence=low,在 unreadable_parts 里说明
```
