# v0.4 — Northstar Loop Implementation Plan

## 0. 关键安全原则

1. **隔离分支**：所有改动在 `v0.4-northstar-loop`。`main` (v0.3 RAG chatbot) 在 WAIC 期间不动
2. **每天 push** → Vercel auto-preview URL → partner 增量验证
3. **保留 v0.3 主链能力**：chat / profile / multi-person 全部不破坏
4. **WAIC 期间** (4/27-4/30) 仅做不影响主链的功能（D1 Memory schema / D2 加补剂入口）；Reminder 等触达类延后到 5/1+

## D1 · MemoryEvent schema + store + 时间轴

### 文件
- `src/lib/memory/types.ts` — MemoryEvent type
- `src/lib/memory/eventStore.ts` — zustand persist (key=`vitame-memory-v1`)
- `src/lib/memory/eventWriter.ts` — 写 event 的辅助函数（chat onFinish / feedback submit / reminder ack 时调用）
- `src/app/memory/page.tsx` — 时间轴 UI
- `src/components/memory/MemoryTimeline.tsx`
- `src/components/memory/EventCard.tsx`

### Schema (北极星 §5)
```ts
export type EventType = 'verify' | 'reminder' | 'feedback' | 'observation' | 'correction';
export type PrivacyMode = 'local' | 'pseudonymous' | 'cloud_memory';

export interface MemoryEvent {
  eventId: string;          // nanoid
  occurredAt: string;       // ISO
  eventType: EventType;
  personId: string;         // 关联到哪个 person
  entityRefs: string[];     // 涉及的 supplement / drug / condition slug
  userText?: string;        // 用户原话（如果有）
  agentText?: string;       // agent 输出（如果有，verify/observation 用）
  tags: string[];           // 自由打标
  privacyMode: PrivacyMode; // P0 全部 'local'
  metadata?: Record<string, unknown>;
}
```

### 自动写 event 触发点
- chat 流结束 + extract delta 后 → 写 `verify` event（含 user query / agent reply summary / entityRefs from delta）
- D3 feedback submit → 写 `feedback` event
- D4 reminder ack → 写 `reminder` event
- D7 hermit 输出 → 写 `observation` event
- 用户在 /memory 页编辑 → 写 `correction` event

### UI
- `/memory` 路由：左 person tabs（沿用 profile 的设计）+ 右 timeline
- 时间轴按天分组，每个 event 一张 EventCard
- EventCard 显示：icon (按 eventType) + 时间 + 简要内容 + entityRefs chip + 展开看详情

## D2 · 加补剂入口 + currentSupplements 字段

### profile schema 加字段
```ts
interface Person {
  // ...existing fields
  currentSupplements: ProfileSupplement[];  // 新加
}

interface ProfileSupplement {
  slug?: string;
  mention: string;       // 用户原话 "汤臣倍健 鱼油 1000mg"
  brand?: string;
  dosage?: string;       // "1000mg/天"
```

### chat extractor 升级
extractor prompt 加新规则：
- "用户主动陈述'我在吃 X 补剂'" → `newSupplements: [{mention:'X'}]`
- 跟 medications 严格区分（医方处方 vs 自购保健品）

### /profile UI
新增 section "正在吃的补剂"，跟 medications 平级但视觉区分（用 stream 蓝色 chip 而非默认）：
- 每条 supplement 显示 brand + dosage + schedule
- 加一个"＋ 添加补剂"内联表单
- 点条目 → /memory?filter=supplement:slug 看相关 events

## D3 · Feedback Ritual

### 文件
- `src/lib/feedback/triggerRule.ts` — 决定何时弹 prompt
- `src/components/feedback/FeedbackPrompt.tsx` — 模态
- `src/lib/feedback/store.ts` — 记录"上次反馈时间"per supplement

### 触发规则（北极星 §4 默认低频）
- 用户访问 chat / profile 时检查
- 如果 currentSupplements 中有任何一款超过 24h 没 feedback event → 触发
- 同一访问 session 最多弹 1 次（避免骚扰）
- 用户连续 3 次跳过 → 频率降到 72h

### Prompt 4 个 variant（北极星 §4）
- "今天吃了 {supplement} 吗？"（吃了 / 没吃 / 跳过）
- "感觉怎么样？"（变好 / 没感觉 / 不舒服 / 跳过）
- "想跳过这次反馈吗？"（跳过 / 提醒我）
- "要调整提醒时间吗？"（早 / 晚 / 不变 / 跳过）

随机选一个，避免每次都问同一个。

### 严重异常关键词硬拦截
如果用户回"不舒服"+ 自由文本里出现 `'剧痛|呼吸困难|出血|过敏|晕倒|抽搐'` → 不让 LLM 推断病因，直接显示"建议立即咨询医生或药师，必要时拨打急救电话"

## D4 · Reminder（in-app banner）

### 文件
- `src/lib/reminder/types.ts` — ReminderRule
- `src/lib/reminder/store.ts` — zustand persist
- `src/lib/reminder/scheduler.ts` — 决定下一次触发时间
- `src/components/reminder/ReminderBanner.tsx` — 顶部 banner
- `src/components/reminder/ReminderRuleEditor.tsx` — /profile 里编辑规则

### Schema
```ts
interface ReminderRule {
  ruleId: string;
  personId: string;
  supplementSlug: string;   // 关联 currentSupplements
  timeOfDay: string;        // "08:00"
  daysOfWeek: number[];     // [1,2,3,4,5,6,7]
  paused: boolean;
  frequencyMultiplier: number; // 1.0=正常, 0.5=降频后
  lastTriggeredAt?: string;
  consecutiveSkips: number;
}
```

### 触发逻辑
- 用户访问任何页面时，scheduler 检查所有 active rules
- 找到"应触发但还没 ack"的 rule → 显示 banner
- banner 内容："该吃 {supplement} 了，吃了吗？" + [吃了 / 跳过 / 改时间]
- ack 后写 `reminder` event + 更新 lastTriggeredAt

### 自动降频规则
连续 3 次 skip → frequencyMultiplier 0.5（每两天一次）
连续 5 次 skip → frequencyMultiplier 0.25 + 弹 prompt 问要不要暂停

### ⚠️ 不依赖浏览器 push notification
P1 仅做 in-app banner（用户访问时显示）。push notification 涉及 PWA + service worker + 权限弹窗，老用户复杂度太高，留给 P2。

## D6-7 · Hermit Agent MVP

### 文件
- `src/app/api/hermit/cron/route.ts` — Vercel Cron 入口
- `vercel.json` — schedule "0 8 * * 1"（每周一早 8 点）
- `src/lib/hermit/analyzer.ts` — 后台 LLM call 分析 events
- `src/lib/hermit/types.ts` — ObservationCard
- `src/components/hermit/ObservationCard.tsx` — 时间轴里的特殊卡片
- `src/lib/hermit/userActions.ts` — accept / dismiss / snooze

### Hermit 周期分析逻辑
```
Cron 触发 → 后台拉所有 person 的近 7 天 MemoryEvent
  → 对每个 person 跑一次 LLM call
  → prompt 含全部 events JSON + 北极星 §6 4 类输出（pattern / recheck / 调整 / 补字段）
  → LLM 返回 ObservationCard[]
  → 写到 eventStore（eventType=observation）
  → 用户下次访问 /memory 看到红点 + observation cards
```

### LLM prompt 严格约束（北极星 §6 不能列表）
- 不诊断疾病
- 不做因果医学归因
- 不自动改方案
- 输出格式："我观察到你 {time-range} {N 次} {action}，{是否要 X}？"
- 必须带具体 events 引用（让用户知道为什么）

### Observation card UI
- 顶部 sprout icon（温和）
- "Hermit 观察"标题 + 时间
- 内容：观察 + 提案
- 操作：[接受提案] [忽略] [稍后再说]
- 点"接受" → 执行（如调整 reminder time）+ 写 correction event
- 点"忽略" → 隐藏 card + 写 correction event
- "稍后再说" → 7 天后再显示

### Vercel Cron 配置
```json
// vercel.json
{
  "crons": [{
    "path": "/api/hermit/cron",
    "schedule": "0 0 * * 1"
  }]
}
```
Hobby 免费版仅 1 daily cron。每周一次足够 MVP。
