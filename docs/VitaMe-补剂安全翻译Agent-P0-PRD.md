# VitaMe — 补剂安全翻译 Agent P0 PRD

> 日期：2026-04-17
> 状态：Draft
> 基于：`VitaMe-补剂安全翻译Agent-定位.md`
> 目的：明确 P0 版本要解决的问题、范围、成功标准和需求边界

---

## 1. 执行摘要

VitaMe P0 聚焦一个清晰问题：

**用户在购买或服用补剂前，不知道它是否适合自己的体质、病史和当前用药。**

用户真正缺少的不是更多信息，而是把专业补剂知识翻译成具体安全决策的能力。

因此，P0 不做泛健康顾问，也不做长期症状陪伴，而是做一个：

**以排雷为入口、以理解和决策为价值交付的补剂安全翻译 Agent。**

---

## 2. 为什么做

### 背景

- 普通用户看得见成分表，但看不懂成分形式、吸收率和适用体质
- 用户知道自己有结石、胃溃疡、高血压等标签，但不知道这会影响哪些补剂选择
- 用户能从视频、课程、社媒获得大量碎片知识，却很难映射到“这瓶能不能吃”
- ChatGPT 可以给一次回答，但很难形成可复查、可沉淀、可追溯的安全决策记录

### Why Now

- 大厂主动回避保健品反向建议和安全判断，市场存在空白
- 规则引擎 + 公共数据库 + LLM 解释层已经足够支持最小可用产品
- 团队当前更需要一个尖锐、可信、易演示的产品楔子，而不是更大的健康平台叙事

---

## 3. 产品目标

### 产品目标

1. 帮用户在购买或服用补剂前做出更安全的决策
2. 帮用户理解“为什么不适合我”，而不是只看到一个风险颜色
3. 用结构化结果沉淀个人或家人的安全档案，为后续复查打基础

### 业务目标

1. 建立 VitaMe 在“补剂安全判断”这一垂直场景的信任
2. 找到一个足够尖锐、容易口碑传播和演示的产品切口
3. 为后续“家庭安全管理层”预留升级空间

---

## 4. 目标用户

### Primary User

**25-40 岁、有补剂购买和筛选需求的决策型用户**

包含两类重叠人群：

1. **给自己买的人**
   - 有体质标签、病史或轻慢病风险
   - 会主动研究补剂，但不能确定自己有没有买对

2. **给父母买的人**
   - 希望补充营养，但担心和父母现有用药或病史冲突

### 当前不优先

- 想要疾病诊断的用户
- 以运动营养和增肌为主的人群
- 只消费健康内容、不进行具体购买或服用决策的人

---

## 5. 核心 Job To Be Done

**当我准备购买、开始服用、替换或追加一款补剂时，我想快速知道它对我是否安全、为什么，以及我应该如何调整决策。**

---

## 6. 成功标准

| ID | 指标 | P0 目标 |
|----|------|---------|
| SC-1 | 首次查询完成率 | >= 80% 用户完成一次查询并看到结果 |
| SC-2 | 结果保存率 | >= 40% 用户在首次查询后保存到档案 |
| SC-3 | 解释理解率 | 访谈中 >= 70% 用户能复述“为什么不适合我” |
| SC-4 | 二次回访率 | 30 天内 >= 25% 用户完成第二次查询或复查 |
| SC-5 | 主观信任度 | 结果可信度评分 >= 4/5 |

---

## 7. 核心场景

### 场景 A：准备买，但不确定适不适合我

用户在社媒、电商或课程里看到某款补剂，想知道：

- 这是不是我该买的
- 它和我的体质、病史、用药有没有冲突
- 商家宣传是不是在偷换概念

### 场景 B：已经买了，但怀疑买错了

用户已经下单或开始服用，想快速确认：

- 现在继续吃有没有风险
- 风险来自成分形式、病史还是当前用药

### 场景 C：想给父母买，但不敢凭感觉买

用户替父母决策，最关心：

- 会不会和现有处方药冲突
- 有没有明显不适合老年慢病体质的地方

---

## 8. 用户故事

1. 作为一个结石体质用户，我希望知道维生素 D、钙和维生素 C 对我是否存在额外风险，这样我不会因为“补营养”反而加重问题。
2. 作为一个正在服用降压药的人，我希望在下单鱼油前先查清交互风险，这样我不用自己去翻论文。
3. 作为一个给父母买补剂的子女，我希望能保存家人的用药和病史，下次新增一款产品时直接复查，而不是重新输入全部信息。
4. 作为一个补剂知识薄弱的普通用户，我希望看懂“氧化镁”和“甘氨酸镁”的区别，这样我能分辨营销话术和真实差异。

## 9. 功能需求

### FR-1 查询入口

- 用户可以输入补剂名称、成分名称或药物名称发起查询
- 系统支持常见商品名到标准成分名的映射

### FR-2 关键风险采集

- 系统通过 2-4 个有限问题采集影响结果的关键上下文
- 采集项包括：病史标签、当前用药、过敏或特殊风险标签
- 不为凑信息而询问无关症状

### FR-3 安全判断引擎

- 对补剂 × 药物 × 病史进行结构化检查
- 结果必须落入红黄灰绿四级之一

### FR-4 安全翻译解释

- 每条结果必须包含“风险原因”解释
- 涉及成分形式差异时，必须用普通语言解释该差异
- 解释必须可直接支持购买或服用决策

### FR-5 证据透明

- 每条结果必须附带来源类型
- 当证据不足时，系统必须明确标注“有限证据”或“暂无数据”

### FR-6 风险规避建议

- 对红黄结果，系统需给出规避方向
- 允许给出“更安全的替代方向”
- 不输出明确诊断和处方性用语

### FR-7 结果保存与复查

- 用户可将本次结果保存到个人或家人档案
- 已保存档案支持后续新增一项时快速复查

---

## 10. 非功能要求

### 合规

- 不使用“治疗”“治愈”“处方”类表述
- 每次输出强制展示免责声明

### 可靠性

- 高风险规则优先走确定性逻辑，不交给 LLM 自由发挥
- LLM 主要负责解释，不负责创造禁忌规则

### 性能

- 首次结果返回控制在 30 秒内
- 常见查询场景优先缓存或预计算

### 可扩展性

- 产品、成分、药物、病史应采用结构化模型
- 为未来家庭档案、复查、体检关联预留数据模型

---

## 11. P0 明确不做

1. 不做疾病诊断
2. 不做完整症状问诊
3. 不做长期症状打卡和每日陪伴
4. 不做泛生活方式方案
5. 不做电商导购和 CPS 佣金
6. 不做中成药全量覆盖
7. 不做体检 OCR
8. 不做家庭多成员复杂协作

---

## 12. 技术边界

P0 推荐架构：

- 前端：手机 H5 单页体验
- 后端：确定性规则引擎 + 数据源查询 + LLM 解释层
- 数据源：优先 SUPP.AI、DDInter、DSLD
- 输出机制：规则先判定，LLM 后解释

---

## 13. 风险与应对

### 风险 1：产品被做成单次检查工具

应对：

- 首次查询后强制引导保存到档案
- 明确设计“新增一项即可复查全部”的体验

### 风险 2：产品被误解为医疗诊断工具

应对：

- 首页、结果页、对话中持续强化边界
- 避免问诊式大入口，聚焦补剂安全决策

### 风险 3：解释层太弱，用户只得到一个颜色

应对：

- 安全翻译层是 P0，不是附加项
- 每条结果都要回答“为什么”

### 风险 4：范围膨胀回到泛健康顾问

应对：

- 禁止把长期症状追踪、疗法推荐、生活方式方案塞进 P0
- 每次需求评审都用一句话定位做边界检查

---

## 14. 后续路线

### P1

- 家庭档案和快速复查做实
- 替代方向建议更结构化
- 增加复购前提醒、换药复查提醒

### P2

- 纳入体检异常项对补剂安全的影响
- 增加服用时间/顺序建议
- 支持家庭协作补全档案

### P3

- 从“补剂安全翻译”升级到“家庭安全管理层”
- 而不是跳成“泛健康顾问”

---

## 15. 一句话结论

**P0 的任务不是做大，而是把“补剂安全翻译”这个场景做到足够可信、可懂、可复查。**
