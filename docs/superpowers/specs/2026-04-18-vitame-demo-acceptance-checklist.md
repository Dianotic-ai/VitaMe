# VitaMe P0 Demo 验收 Checklist(人话版)

> 用法:Claude Code 告诉你"写完了"之后,**你自己打开浏览器**,按下面的清单一条一条试,勾对号。
> 不用读代码,不用读 plan,不用读 spec。只看浏览器里显示什么。

---

## 这个产品做什么(一段话)

用户输入一个补剂 → 回答 2–4 个关键问题 → 得到**红黄灰绿 + 为什么 + 该怎么办 + 存档**。就这。

不做疾病诊断,不做长期打卡,不做电商导购。

---

## Demo 核心体验(用户眼睛能看到的 3 步)

1. **首页**:一个输入框 + 一句"买补剂前,先查它适不适合你的体质"。Placeholder 写自然语言例 "问我吧,比如:我妈在吃华法林,能吃辅酶 Q10 吗？"（v2.8 取消 v2 关键词输入）
2. **意图识别 + 追问（v2.8）**：系统先听懂"你想问什么"——识别成 7 类意图之一。如果信息够 → 直接出结果；如果缺关键 slot → 出最多 2 轮 ClarifyBubble（每个气泡 ≤4 选项 + 永远附"其他（自己说）"入口）。**不**再硬性 4 题问答。
3. **结果页**:色块(红/黄/灰/绿)+ 原因 + 建议 + 证据来源 + "保存到档案"按钮

## 12 条验收 Checklist(7 必保 + 4 体验 + 1 主场景亮点)

### 🚨 必保红线(7 条,任一翻车 = 停下来修,别急着 Demo)

#### 规则引擎判对(3 条)

- [ ] **1. 华法林 + 辅酶 Q10 → 红色**
  - 输入:`我妈在吃华法林,能吃辅酶 Q10 吗?`
  - 期望:色块**红色** + 明确写"不建议自行服用"
  - 顶部必有强警示"建议立即咨询医生或药师"
  - ❌ 翻车信号:出了黄色、绿色,或没有就医提示

- [ ] **2. 孕期 + 维 D → 黄色 + 强免责**
  - 输入:`我怀孕了,维 D 要吃多少?`
  - 期望:色块**黄色**(不能是红色,也不能是绿色)
  - 免责声明必须追加一句"孕期补剂必须经产科医生确认"
  - ❌ 翻车信号:出绿色"安全可吃",或没有孕期强免责

- [ ] **3. 胃溃疡 + 氧化镁 → 黄色 + 推荐换形式**
  - 输入:`我胃溃疡,氧化镁能吃吗?`
  - 期望:色块**黄色**
  - 建议必须含"甘氨酸镁"或"苏糖酸镁"作为更安全替代
  - ❌ 翻车信号:只说"不要吃",没给替代方向

#### 合规红线必守(3 条)

- [ ] **4. 每个结果页底都有 disclaimer**
  - 翻 3 个不同场景的结果页,底部都有"VitaMe 提供补剂安全信息和决策辅助,不提供疾病诊断..."这段话
  - ❌ 翻车信号:任一结果页没 disclaimer

- [ ] **5. 禁词全不出现**
  - 随便翻几个结果页的文本
  - **绝对不能**出现:`治疗` `治愈` `处方` `诊断` `药效` `特效` `根治`
  - ❌ 翻车信号:出现任何一个词

- [ ] **6. grading 语义对（v2.8 新增 / 替代旧第 6 条）**
  - 6a — **未收录 → 灰色（"未收录"标签）**：输入一个数据库查不到的冷门补剂（如 `黄金燕窝肽`）。期望 RiskBadge 灰 + 文案"未收录"。
  - 6b — **已收录 + 无禁忌 → 绿色（"未见风险"）**：输入 `健康成年人,无慢病,无吃药,能吃维生素 C 吗？`。期望 RiskBadge 绿 + 文案"未见风险"（而**不是**灰）。
  - 6c — 没在结果页看到 LLM 自由发挥的"安全可吃"结论。
  - ❌ 翻车信号 1：6a 出绿色（把"未收录"误判为"安全"）
  - ❌ 翻车信号 2：6b 出灰色（把"无禁忌"误判为"未收录"）
  - ❌ 翻车信号 3：胡编"安全可吃"结论

- [ ] **7. Demo Banner 正确挂出(D2 新增,CLAUDE.md §11.11)**
  - 触发:任一硬编码 `Contraindication` 命中且未经临床资质审核(P0 阶段 50 条规则默认全部未审)
  - 期望:结果页顶部显示横幅"本 Demo 为原型展示，禁忌规则由产品团队基于 NIH ODS（美国国立卫生研究院膳食补充剂办公室）、Linus Pauling Institute（美国俄勒冈州立大学微量营养素信息中心）、SUPP.AI（美国 补剂-药物相互作用数据库）、中国营养学会 DRIs（中国居民膳食营养素参考摄入量）等公开权威数据整理，尚未经执业药师临床复核，不构成医疗建议。"
  - 验证:翻第 1/2/3 条的结果页(华法林 + Q10 / 孕期 + 维D / 胃溃疡 + 氧化镁),均应挂出 Banner
  - 额外验证(合规审计):在服务端 `audit/*.jsonl` 最新一条中 grep `"demo_banner_hit":true` 应命中
  - ❌ 翻车信号 1:结果页无 Banner(视觉漏挂)
  - ❌ 翻车信号 2:Banner 只在前端计算,audit log 里 `demo_banner_hit = false`(违反 §11.10,合规严重)

### 💚 体验验收(7 条,达到 5 条即可)

- [ ] **8. 不做无关追问**
  - 输入:`鱼油+维D+镁+益生菌+维B+维C+D3`(7 种补剂一起)
  - 期望:系统最多 2 轮 clarify（v2.8）+ 每轮只问与判断直接相关的（用药 / 慢病 / 过敏 / 特殊群体）
  - ❌ 不该问:"睡眠怎么样" / "食欲如何" / "运动习惯" / "每天喝几杯咖啡"

- [ ] **9. 成分形式差异讲得懂**
  - 输入:`镁`(只输入镁,不带形式)
  - 期望:看到 4 种形式对比(氧化镁 / 柠檬酸镁 / 甘氨酸镁 / 苏糖酸镁)
  - 且按目的推荐(睡眠 → 甘氨酸镁 / 便秘 → 柠檬酸镁 / 认知 → 苏糖酸镁)
  - ❌ 翻车信号:只说"补镁有助健康",没拆形式

- [ ] **10. 档案保存 + 复查省输入(这是产品核心价值)**
  - 第 1 步:输入 `我妈肝炎+脂肪肝+apoe4,能吃鱼油吗?` → 保存到"妈妈"档案
  - 第 2 步:回档案页,在"妈妈"下点"新增一项" → 输入 `钙片`
  - 期望:**系统不再问**"有无肝病"/"有无基因检测",直接出复查结果
  - ❌ 翻车信号:又让你重输一遍病史

- [ ] **11. 60 秒能拿到结果（v2.8 调整 — 30s → 60s）**
  - 从首页输入 → 看到色块结果
  - 期望:整个过程 ≤ 60 秒(手机 4G 网络下，含 ≤2 轮 clarify)
  - ❌ 翻车信号:转圈超过 90 秒
  - 决策依据：稳定可靠 > 速度极致；Demo 视频可剪（D5 决策）

- [ ] **12. 意图识别走通 4 个 P0 handler（v2.8 新增）**
  - 12a — `product_safety_check`：输入 `我妈在吃华法林，能吃辅酶 Q10 吗？` → 走标准红黄灰绿流程
  - 12b — `symptom_goal_query`：输入 `我最近睡不好,有什么补剂能帮上吗？` → 出 `SymptomCandidateList`（≤5 候选 + 必带"不构成医疗建议"注脚 + 每条带"查这个安不安全 →"按钮触发二次安全核查）。**禁止**直接出 RiskBadge（必须二次点击进 product_safety_check）。
  - 12c — `ingredient_translation`：输入 `Magnesium Bisglycinate 是什么？` → 出成分翻译卡（中英对照 + 形式简介），**不**自动判风险。
  - 12d — `unclear`：输入 `你好啊` → 出 `IntentFallbackForm`（3 条引导例句 + 输入框）。**禁止**报错栈 / `intent_llm_timeout` / raw LLM 字串。
  - ❌ 翻车信号 1：12a–12c 任一 handler 走错（intent 识别错）
  - ❌ 翻车信号 2：12b 直接出 RiskBadge 而没经二次确认（违反 §11.14）
  - ❌ 翻车信号 3：12d 漏出技术错误信息（违反 §11.6 / §11.13）

- [ ] **13. ClarifyBubble 单题 + ≤4 选项 + "其他"入口（v2.8 新增）**
  - 触发:输入 `维生素 D` 这种缺人群信息的查询
  - 期望:出现一个气泡（不是 4 题列表），≤4 个选项 button + 永远附 "其他（自己说）" ghost button
  - 期望:气泡下面有小字 "为什么问？"，点开能解释这条 clarify 是为啥（合规可解释性）
  - 期望:整次会话 ≤2 个 clarify 气泡。第 3 轮还 unclear → 强制 fallback 到 IntentFallbackForm
  - ❌ 翻车信号 1：一次性出 4 个气泡（违反 spec）
  - ❌ 翻车信号 2：选项 ≥5（违反 schema）
  - ❌ 翻车信号 3：没有 "其他" 入口（用户走死胡同）

- [ ] **14. clarifyingQuestion schema 不出 risk 字段（v2.8 红线 §11.13）**
  - 验证：在 DevTools Network 查 `/api/intent` 响应；`clarifyingQuestion` 对象**只**含 `question` + `choices`
  - ❌ 翻车信号：响应里出现 `level` / `safe` / `dangerous` / `riskScore` 等任一字段（说明 LLM 越权 + Zod `.strict()` 没拦住，合规严重）

### 🎥 主场景亮点(1 条,亮点档才需要;保底档可降级为纯文字 Demo)

- [ ] **15. OCR 拍照输入(主场景 A)**
  - 输入:上传一张 Now Foods Magnesium Glycinate 200mg 瓶子照片(Demo 提供标准图)
  - 期望:OCR 识别出 brand / product_name / 成分表,自动填入追问流程
  - 兜底:识别失败时,UI 必须有"手动输入"入口,不能报错卡死
  - ❌ 翻车信号:识别失败 + 无手动输入兜底 = 用户走不下去
  - 降级规则:若 D5 OCR 联调未过,本条自动降级为"纯文字 Demo",不计翻车

## 验收时最常见的 3 种翻车模式

1. **规则引擎错了**(最严重,合规问题)
   - 症状:华法林 + Q10 出了黄色或绿色
   - 原因:硬编码禁忌规则没加,或被 LLM 覆盖了
   - 修法:把这条规则截图发 Claude Code,让它检查 `contraindications` 数据表

2. **LLM 飘了**(中等,合规问题)
   - 症状:结果页里出现了"治疗"/"处方"这种词
   - 原因:禁词过滤没拦住,或直接绕过了过滤层
   - 修法:截图给 Claude Code,让它检查 GuardrailFilter 是否全量拦截

3. **免责声明漏了**(中等,合规问题)
   - 症状:某个结果页底部没 disclaimer
   - 原因:那条 API 路径没挂合规中间件
   - 修法:截图给 Claude Code,让它检查所有 API response 是否都走 DisclaimerInjector

## 怎么反馈翻车给 Claude Code

**别光说"不对",要说清三件事:**

1. **输入了什么**(原话)
2. **期望看到什么**(对照 Checklist 里的"期望")
3. **实际看到什么**(截图 + 文字转述)

> 验收 Checklist 第 1 条翻车。
