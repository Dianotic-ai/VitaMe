# 开源项目复用评估（ChatGPT 推荐校对）

> 触发：2026-04-23 Kevin 提供了 ChatGPT 对话原文，推荐了 11 个健康/营养/补剂 GitHub 项目
> 方法：按 VitaMe 实际需求重排 → 对 Top 4 候选用 `gh repo view` 核实许可证、活跃度、体量
> 结论：**ChatGPT 推荐存在两个严重错误**；可直接复用的只有 VITA-LOG；药物-补剂交互数据**仍需自建**

---

## 一、方法论

### 1.1 评估维度

| 维度 | 检查项 | 工具 |
|---|---|---|
| **定位匹配度** | 项目逻辑是否与 VitaMe（给定补剂→判断是否安全+提醒）对齐 | 人工判断 |
| **法律可用性** | 许可证类型（Apache/MIT 优 vs GPL/AGPL/无license 谨慎） | `gh repo view --json licenseInfo` |
| **活跃度** | 最近推送时间、star/fork 数 | `gh repo view --json pushedAt,stargazerCount` |
| **实际体量** | diskUsage 能反映是否有真实数据集 | `gh repo view --json diskUsage` |
| **可迁移性** | 语言/框架是否能在 VitaMe 技术栈（H5/TS/DeepSeek）内重用 | primaryLanguage 检查 |

### 1.2 重要原则

**不能基于"项目定位描述"做采用决策**。必须核对：
1. 许可证（否则合规风险）
2. 实际代码量（否则可能是空壳）
3. 最近活跃度（死项目用起来有风险）

ChatGPT 的推荐是基于项目 README 的定位描述做的，**它并没有核对以上三项**。这是本评估报告的核心价值。

## 三、核心错位：ChatGPT 误判了 VitaMe 的产品逻辑

| 维度 | ChatGPT 的理解 | VitaMe 实际定位 |
|---|---|---|
| 产品逻辑 | "你需要什么补剂" → 推荐 | "给定这个补剂 → 判断是否安全 + 管理服用" |
| 核心能力 | 营养缺口分析 + 推荐 | 交叉禁忌判断 + 替代建议 + 提醒闭环 |
| 数据需求 | USDA 营养数据库 | 禁忌规则库 + 药物×补剂交互表 + 品牌检测库 |

**后果**：ChatGPT 的"第一梯队（补剂问答）"对我们价值反而低；真正有价值的是**第三梯队的药物交互 + VITA-LOG 追踪**。

## 六、ChatGPT 推荐的两个严重误导（纠正记录）

### 误导 1：把 `LLM-Drug-Interaction-Checker` 列为高价值辅助模块

**ChatGPT 原话**："对'保健品 agent'，药物/补剂相互作用检查几乎是必备模块"——判断正确。

**但推荐的 repo 错了**：22KB 的 Streamlit 小项目根本不具备生产可用的数据。ChatGPT 只看了 description，没看 diskUsage。

**纠正**：VitaMe 的药物-补剂交互规则库**必须从权威数据源自建**。候选数据源：
- **DrugBank**（学术许可，商业需付费）
- **NIH Office of Dietary Supplements (ODS)** — 美国国立卫生院官方，免费
- **MedlinePlus Supplements** — NIH 消费者版
- **Natural Medicines Database** — 商业订阅，最权威
- **Memorial Sloan Kettering Herb-Drug Interactions** — 学术免费

### 误导 2：把 `vitamins-and-supplements-api` 列为"知识/工具层"

**ChatGPT 原话**："很适合做 agent 的知识/工具层"——方向正确。

**但体量错了**：109KB = 数十种补剂的简单静态数据，不是"知识库"。

**纠正**：知识层数据源应该是上面列的权威源，或者至少是商业化的 ConsumerLab / Examine.com 订阅数据。`vitamins-and-supplements-api` 最多值 1 小时的"字段命名参考"。

## 八、ChatGPT 推荐的一个正确判断（值得采纳）

它最后说："如果你是想做成'skill'而不是大而全产品，反而建议拆成 4 个 skill：
补剂知识检索、营养缺口判断、相互作用检查、服用计划与追踪。"

**对应到 VitaMe 的映射**（3 对齐，1 不做）：

| ChatGPT 建议的 skill | VitaMe 对应 | 采纳 |
|---|---|---|
| 补剂知识检索 | 规则库 + 成分解释层 | ✅ |
| 营养缺口判断 | — | ❌（违反 P0 "不回答长期调理"） |
| 相互作用检查 | 规则引擎核心 | ✅ |
| 服用计划与追踪 | Reminder + Feedback + Memory | ✅ |

"拆 skill 而非找大而全 repo" 的思路**跟 VitaMe 的工程分层一致**（见 `docs/superpowers/specs/` 的模块化设计）。

## Insights

- **"项目定位匹配"不等于"可用"**。ChatGPT 的推荐全部基于定位描述，它无法评估许可证、数据量、活跃度——而这三项恰恰是决定"能不能真的用"的关键。做开源选型必须用 `gh repo view` 或等价工具做 due diligence，不能信二手描述。
- **健康类商业产品做开源选型时，许可证是第一关卡**。GPL/AGPL/无 License 的项目都有传染性或法律风险。这意味着 VitaMe 在规则库 / 数据源这块，大概率要**自建或走商业订阅**（NIH ODS 免费但数据粒度有限；Natural Medicines Database 付费但权威）。
- **ChatGPT 最有价值的输出是最后那段建议**（拆 skill 而不是找大而全 repo），而不是具体的 repo 列表。这跟它的训练分布有关：它擅长做"框架性建议"，但具体 repo 级别的推荐经常因为 training cutoff + 项目死亡率而不可靠。**以后类似研究任务，让 Agent 用 `gh search` 现场搜 + `gh repo view` 现场核实比问 LLM 更可靠。**
