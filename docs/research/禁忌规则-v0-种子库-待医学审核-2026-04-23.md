# VitaMe 禁忌规则 v0 — 种子库

> 来源：`docs/research/gemini-health-consultation.md` 14 轮对话
> 目的：为 P0 硬编码规则引擎提供**种子库**——医学博士审核前的 draft 版本
> 规模：32 条规则，覆盖 8 个病史 / 成分类别
> 状态：**Draft**，需医学检测博士 + 护士二审

---

## 字段定义

| 字段 | 含义 |
|---|---|
| **ID** | 规则唯一标识，格式 `R-类别-序号` |
| **条件** | 触发规则的 病史 × 成分 / 产品特性 交叉 |
| **等级** | ❌ 禁 / ⚠️ 警 / ✅ 推（推荐） |
| **机制** | 1-2 句说明为什么（供"点开展开原因"使用） |
| **Tier** | 触发所需的最低档位：Tier 0（通用/纯商品属性）/ Tier 1（人群大类）/ Tier 2（具体病史） |
| **引擎** | 硬编码 / LLM 辅助（硬编码 = 不可被 Prompt 覆盖） |

### 等级 UI 对应

- ❌ 禁 = 红色拦截弹窗，强制展示"不建议" + 替代方向
- ⚠️ 警 = 黄色提醒，展示注意事项 + 建议就医询问
- ✅ 推 = 绿色建议，展示"适合" + 机制说明

### Tier 档位触发策略

- **Tier 0**：不需要用户任何信息，基于商品本身即可触发（剂量阈值、商品属性）
- **Tier 1**：需要用户回答 1 个多选问题（性别 / 年龄段 / 3 个病史大类）
- **Tier 2**：需要具体的病史字段（ICD 编码粒度）

## 规则清单

### A. 结石 / 泌尿系统（6 条）

| ID | 条件 | 等级 | 机制 | Tier | 引擎 |
|---|---|---|---|---|---|
| R-KD-01 | 结石体质 + VC ≥ 1000mg/d | ❌ 禁 | VC 代谢为草酸 → 加剧草酸钙结石 | Tier 1 | 硬编码 |
| R-KD-02 | 结石体质 + 单独钙片（非 D3+K2 组合） | ⚠️ 警 | 盲目补钙加重肾脏排泄负担；钙应通过奶制品/绿叶蔬菜获取 | Tier 1 | 硬编码 |
| R-KD-03 | 结石体质 + D3 > 2000 IU 且无 K2 | ❌ 禁 | K2 缺席时钙可能沉积在肾/血管而非骨骼；结石体质风险翻倍 | Tier 1 | 硬编码 |
| R-KD-04 | 结石体质（推荐） | ✅ 推 柠檬酸镁 | 柠檬酸根碱化尿液、抑制草酸钙结晶；同时镁拮抗钙沉积 | Tier 1 | 硬编码 |

## 规则触发档次统计

| Tier | 规则数 | 说明 |
|---|---|---|
| **Tier 0**（纯商品属性 / 通用） | 6 | 用户 0 输入即可触发，撑起 Verify 的"通用人群模式" |
| **Tier 1**（1 个多选问题可判） | 6 | 冷启动第一问升档后可用 |
| **Tier 2**（具体病史） | 20 | 深度匹配，Feedback Ritual 或主动问答中逐步解锁 |

**关键观察**：Tier 0 规则只有 6 条——这是冷启动"通用人群模式"能覆盖的安全底线。要让 Tier 0 真正有用，需要**规则数量不足 + 文案引导升档**的组合拳。用户看到 "⚠️ 含氧化镁可能腹泻 — 告诉我你是否容易腹泻可以更准判断" 就是一次自然的 Tier 1 升级钩子。

## 引擎实现建议（工程）

### 数据结构

```typescript
type Rule = {
  id: string;                  // "R-KD-01"
  conditions: {
    user_profile?: string[];   // ["N20.0"] 即结石病史（ICD-10）
    product_attrs?: string[];  // ["ingredient:vitamin_c", "dose_mg:>=1000"]
    context?: string[];        // 可选：用药、饮食、场景
  };
  severity: "deny" | "warn" | "recommend" | "tip";
  mechanism: string;           // 机制说明（面向用户的短文）
  tier: 0 | 1 | 2;
  source: "hardcoded" | "llm_assisted";
  review_status: "draft" | "doctor_reviewed";
};
```

### 评估策略

1. **每次 Verify 输入一个 SKU** → 解析 SKU → 得到 `product_attrs`
2. **遍历全量规则**，按 `Tier ≤ 当前用户档位` 过滤
3. **匹配条件**：`user_profile ⊇ rule.conditions.user_profile` AND `product_attrs ⊇ rule.conditions.product_attrs`
4. **按严重度排序展示**：deny > warn > recommend > tip
5. **遇到 Tier 1/2 规则所需字段缺失**：生成升档钩子（"告诉我你是否 X，可让这条建议变准"）

### LLM 的位置

LLM **不参与规则判断**，只做两件事：

1. **商品名归一化**：`"NOW Foods Ultra Omega-3"` → `{ ingredient: "omega-3", form: "enteric-coated", epa_mg: 500, dha_mg: 250 }`
2. **机制解释的个性化措辞**：把机制说明"翻译"成给具体用户看的一句话（如"对你这种经常熬夜的情况，Q10 尤其能..."）

## 风险与局限

1. **Draft 级别**：未经医学方审核，**不可直接上线**。仅供工程骨架搭建使用。
2. **剂量阈值粗放**：部分阈值（如 VC 1000mg）引用通用指南，个体差异（体重、肾功能）未细分。医学方二审时应细化。
3. **中西医混合风险**：现规则几乎全是"西医 + 功能医学"视角。中成药、中药材的禁忌未覆盖，属主动缺省（P0 不做）。
4. **药物-补剂相互作用未覆盖**：如华法林 × 高剂量 VK、SSRI × 圣约翰草。P0 若要支持"当前用药字段"，需补充一份药物-补剂交互表（~100 条）。
5. **规则 ID 的长期维护**：建议规则 ID 一经医学方确认后冻结，后续改动用新 ID + deprecate 旧 ID，保留审计轨迹。

## Insights

- **Tier 0 只有 6 条规则 = 必须用"询问升档"而非"禁言"来体验补齐**。这验证了 SOP v2 里"Tier 0 的输出必须标注`告诉我 X 可让判断变准`"的设计。
- **规则的本质是"条件 → 机制 → 行动"三段论**，不是"if-else"。每条规则都要能"点开看原因"——这是 VitaMe 区别于丁香医生的关键，后者只给结论。
- **硬编码规则 + LLM 语义归一化**的分工让责任可追溯：出事时规则是查得到的代码，LLM 只做"翻译"——这对健康类产品的合规性非常关键。
