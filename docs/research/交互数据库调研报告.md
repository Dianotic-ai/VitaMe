## 二、核心问题回答

### Q1: 仅用公开数据库能构建多大覆盖率的交互检测引擎?

**预估覆盖率：西药DDI 70-80%, 药-补充剂 40-50%, 药-食品 20-30%, 中药交互 <10%**

- **西药DDI**：DDInter(30万条) + RxNav/DrugBank开放版 + TWOSIDES，可覆盖主流处方药的大部分已知交互。
- **药物-补充剂**：SUPP.AI(5.9万条, NLP提取, 含噪声) + DSLD(标签数据) + MSKCC(290种)，覆盖常见西方补充剂，但中药补品几乎空白。
- **药物-食品**：DDInter 2.0 新增857条DFI + FARFOOD(计算预测) + FooDB化合物映射，可做基础框架，但远不够全面。
- **中药**：TCMSP/SymMap/HERB等提供成分-靶点映射，但**不直接提供临床交互数据**。中药与西药的交互数据极度稀缺。

### Q2: 数据质量够不够做消费者产品?

**不够，存在三大风险：**

1. **NLP提取数据含噪声**：SUPP.AI通过AI从文献自动提取，存在误判。TWOSIDES基于统计关联而非因果验证。
2. **覆盖偏差**：研究热门药物/补充剂数据丰富，冷门品种几乎空白。中国特色保健品(如灵芝孢子粉、阿胶)基本无数据。
3. **时效性不足**：TCMSP(2014停更)、TWOSIDES(~2016停更)，部分核心库已过时。

**建议**：公开数据可作为底层知识图谱，但**面向消费者的交互提示必须经过医学专业审核**，至少需要药学背景人员审核高风险交互标注。可采用分级策略：高置信度交互(来自多个数据源交叉验证)自动提示，低置信度交互标注"建议咨询医生"。

### Q3: 最推荐的数据库组合?

**核心引擎层（必选）：**
| 层 | 数据库 | 作用 |
|----|--------|------|
| 药物-药物 | DDInter 2.0 + RxNav API | 30万+条DDI, 免费, 可编程 |
| 药物-补充剂 | SUPP.AI + DSLD | AI提取交互 + 补充剂成分标签 |
| 不良反应 | OpenFDA (FAERS+CAERS) | 免费API, 真实世界安全信号 |
| 化合物映射 | PubChem PUG-REST | 统一化合物ID, 结构匹配 |

**扩展层（推荐）：**
| 层 | 数据库 | 作用 |
|----|--------|------|
| 药物-食品 | FARFOOD + FooDB + DDInter DFI | 药-食交互预测 |
| 中药基础 | HERB + TCMSP + SymMap | 中药成分-靶点知识图谱 |
| 循证补充 | PubMed API + ClinicalTrials.gov | 按需检索最新证据 |

**中国本土层（需爬取/合作）：**
| 层 | 数据库 | 作用 |
|----|--------|------|
| 药品名录 | NMPA数据查询 | 中国批准药品映射 |
| 保健食品 | 蓝帽子特殊食品平台 | 中国保健食品成分 |

### Q4: 能否覆盖"中国处方药 x 西式保健品 x 中药传统补品"?

**目前没有任何单一数据库组合能完整覆盖这三者的交互。** 现实路径如下：

| 交叉场景 | 可用数据 | 覆盖度 | 缺口 |
|----------|---------|--------|------|
| 中国处方药 x 西式保健品 | SUPP.AI + DrugBank(映射通用名) | 30-40% | 中国独有仿制药命名映射 |
| 中国处方药 x 中药补品 | TCMSP靶点 + PubMed文献检索 | <5% | 临床交互证据极少 |
| 西式保健品 x 中药补品 | 几乎无结构化数据 | ~0% | 完全空白 |

**可行方案：**
1. 用RxNorm/DrugBank的ATC编码映射NMPA药品名，打通中-英药物对照
2. 中药部分以HERB+TCMSP的成分-靶点数据为底，用PubMed API实时检索"herb X + drug Y + interaction"文献
3. 用LLM(如Claude)对检索到的文献进行结构化提取，构建增量知识库
4. **MVP阶段建议聚焦**：Top 100常用处方药 x Top 50常见保健品，手动标注交互，覆盖80%用户场景

---

## 三、结论

公开数据库可支撑一个**有限但可用的交互检测MVP**，覆盖西药DDI和常见西方补充剂交互。中国处方药可通过通用名映射部分对接。但**中药与西药/保健品的交互**是全球性数据空白，需要自建数据+LLM文献挖掘+专家审核的组合策略来逐步填充。
