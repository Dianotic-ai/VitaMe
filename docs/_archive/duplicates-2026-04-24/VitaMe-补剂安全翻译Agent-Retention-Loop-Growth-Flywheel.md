# VitaMe — 补剂安全翻译 Agent Retention Loop / Growth Flywheel

> 日期：2026-04-17
> 状态：Draft
> 目的：定义 VitaMe 的回访机制、留存逻辑和增长飞轮

---

## 1. 先定一条原则

VitaMe 不是高频陪伴型产品。

它的自然形态是：

**事件触发型、高价值决策工具。**

所以 P0 不应该追求伪日活，也不应该拿“每天打卡”当留存主轴。

VitaMe 的留存成立方式是：

**一次排雷  
→ 一次理解  
→ 一次保存  
→ 下一次更轻的复查  
→ 形成习惯和家庭协作。**

---

## 2. 当前版本的留存命题

### 留存不是“每天回来聊”

当前定位下，用户回来的原因不是：

- 我今天想聊聊健康
- 我今天想被陪伴
- 我今天想记录症状

而是：

- 我看到一个新补剂，想先查
- 我准备复购，想再确认
- 我换药了，想重新检查
- 我要给父母加一项，想看全局有没有冲突

### 留存的本质

**让第二次决策比第一次决策轻。**

如果 VitaMe 做不到这一点，它就只是一次性工具。

---

## 3. P0 留存闭环

### Loop 1：决策留存闭环

外部事件触发  
→ 用户发起查询  
→ VitaMe 给出风险判断和解释  
→ 用户保存到档案  
→ 下次新增一项时直接复查  
→ 用户形成“先查再买”的习惯

### 这条闭环为什么成立

- 用户第一次得到的是“理解”
- 用户保存下来的不是内容，而是“决策上下文”
- 第二次回来时，输入成本明显下降

### 这条闭环的失败条件

- 保存动作过重
- 档案没有减少输入成本
- 第二次回来仍然像第一次

---

## 4. P0 增长飞轮

### 核心飞轮

高风险决策触发  
→ 30 秒给出明确判断  
→ 结果可理解、可转发  
→ 用户保存到个人 / 家人档案  
→ 下一次新增时快速复查  
→ 用户把结果分享给家庭中的另一个决策者  
→ 家庭档案更完整  
→ 结果更可信、更有复查价值  
→ 更容易继续使用和传播

### 飞轮中的每一环

| 环节 | 用户得到什么 | 产品得到什么 |
|---|---|---|
| 触发 | 一个值得立刻查询的理由 | 高意图流量 |
| 查询 | 一次风险判断 | 行为数据 |
| 解释 | 一次真正理解 | 信任积累 |
| 保存 | 下一次更轻的复查 | 档案沉淀 |
| 复查 | 输入成本下降 | 回访 |
| 分享 | 家庭共识和确认 | 新用户和协作者 |
| 补全 | 更完整的上下文 | 更高质量数据 |

---

## 5. P0 用户为什么回来

### 主要回访原因

1. **新增补剂**
2. **换药**
3. **准备复购**
4. **给父母新增一项**
5. **看到新的推荐内容后想交叉验证**

### 次级回访原因

1. 想把第一次没保存的结果补存进去
2. 想给另一位家庭成员也查一下
3. 想把之前查过的结论分享给别人

### 不是当前版本的回访原因

1. 每日症状跟踪
2. 每周改善报告
3. 被长期陪伴式提醒召回

---

## 6. P0 的增长抓手

### 抓手 1：尖锐入口

首页和结果页必须让用户立刻明白：

**这不是健康内容，而是风险判断。**

### 抓手 2：结果可转发

分享出去的不是广告，而是一个具体结论：

- 我帮妈妈查到一个黄色风险
- 这个鱼油和降压药要注意
- 不是所有镁都适合结石体质

### 抓手 3：家庭协作

最自然的裂变不是“转发给朋友”。

而是：

**把另一个家庭决策者拉进来，一起维护同一份上下文。**

---

## 7. P1 / P2 如何放大飞轮

### P1

- 让档案和快速复查真正做实
- 增加复购前提醒
- 增加换药后复查提醒

### P2

- 增加家庭协作补全档案
- 增加家庭结果分享和确认
- 增加与体检异常项相关的复查触发

### 不建议提前做的留存动作

- 每日健康打卡
- 泛症状追踪
- 为了活跃而活跃的消息召回

---

## 8. 指标体系

### P0 应追的指标

| 指标 | 意义 |
|---|---|
| 首查完成率 | 用户是否真的进入第一价值点 |
| 保存率 | 用户是否愿意沉淀上下文 |
| 30 天二次回访率 | 闭环是否成立 |
| 二次查询平均耗时 | 第二次是否明显更轻 |
| 家庭档案创建率 | 家庭场景是否被激活 |
| 分享率 | 结果是否值得传播 |

### 不建议过早追的指标

| 指标 | 原因 |
|---|---|
| DAU | 对当前产品形态误导很大 |
| 日均使用时长 | 时间长不代表价值高 |
| 每周打开次数 | 事件型产品不应被高频指标绑架 |

---

## 9. 设计约束

1. **保存必须成为结果页的一部分，不是次要入口**
2. **复查必须明显更轻，不然飞轮会断**
3. **分享的内容必须具体，不要做品牌宣传卡**
4. **档案是为复查服务，不是做后台管理系统**
5. **增长文案必须围绕风险和理解，而不是围绕“更健康生活”**

---

## 10. 一句话结论

**VitaMe 的留存，不靠“每天来聊”，而靠“每次要做补剂决策时，先来查，再把这次判断沉淀成下次更轻的决策”。**
