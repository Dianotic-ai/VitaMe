# VitaMe — 补剂安全翻译 Agent Aha Moment Activation Spec

> 日期：2026-04-17
> 状态：Draft
> 目的：定义 P0 的第一价值瞬间、激活路径和可验证标准

---

## 1. 为什么先定义 Aha 和 Activation

VitaMe P0 不是内容产品，也不是陪伴产品。

它能不能成立，不取决于用户有没有逛很久，而取决于用户是否在第一次使用里完成这 3 件事：

1. 看到一个具体风险或差异
2. 真正理解“为什么和我有关”
3. 愿意把这次判断沉淀下来，作为下一次复查的起点

如果这 3 件事没有发生，后面的留存和增长都不会成立。

---

## 2. P0 的 Aha Moment 定义

### 核心定义

**用户第一次明确意识到：同样叫“补剂”，成分形式、体质、病史和当前用药会让“能不能吃”发生变化，而且 VitaMe 能把这个差异解释清楚。**

这不是“我看到一个红黄灰绿颜色”。

这也不是“我看完一段科普”。

它必须是：

**“原来不是补剂都一样，而且我现在知道为什么这款对我不合适/需要注意。”**

### 典型 Aha 文案

- 原来不是补镁都一样
- 原来是我这个体质不适合这种形式
- 原来商家卖点和我的真实适配性不是一回事
- 原来风险不是玄学，是病史 / 用药 / 成分形式共同决定的

### 非 Aha

- 用户只看到颜色，没有理解原因
- 用户只觉得“这像另一个搜索结果页”
- 用户觉得答案和 ChatGPT 没有本质差别

---

## 3. Activation 的定义

### 弱激活

用户完成一次查询，并看到带解释的结果页。

### 强激活

用户完成一次查询后：

1. 能复述“为什么不适合我 / 为什么需要注意”
2. 愿意据此做出购买、停止、替换或复查决策
3. 愿意保存结果到个人或家人档案

### P0 真正追求的激活

**强激活**

因为 VitaMe 不是靠“看过结果”成立，而是靠“理解结果 + 信任结果 + 保存结果”成立。

---

## 4. P0 激活路径

### Activation Flow

首页进入  
→ 输入补剂 / 成分 / 药物  
→ 回答 2-4 个关键问题  
→ 得到红黄灰绿结果  
→ 看到“为什么”解释  
→ 做出一个决策  
→ 保存到档案

### 每一步的用户任务

| 阶段 | 用户在做什么 | 系统必须交付什么 |
|---|---|---|
| Landing | 判断值不值得试 | 一个尖锐承诺：买前先查适不适合你的体质 |
| Input | 输入手上的具体对象 | 低摩擦、可识别的查询入口 |
| Intake | 补齐最少必要上下文 | 只问影响结果的问题，不盘问 |
| Result | 接收风险判断 | 清晰等级 + 原因解释 |
| Decision | 决定停、换、买或复查 | 可执行的下一步 |
| Save | 决定是否沉淀结果 | 明确告诉用户“保存能让下次更轻” |

---

## 5. Activation 的关键断点

### 断点 1：首页承诺不够尖

用户如果看不出“这和普通健康内容有什么不同”，就不会开始。

### 断点 2：输入门槛太高

用户如果一上来就要填太多病史、症状、体检数据，会直接流失。

### 断点 3：结果只有颜色，没有解释

用户看不懂为什么，就不会建立信任，也不会保存。

### 断点 4：没有明确的保存理由

如果“保存到档案”像后台管理动作，用户不会做。

---

## 6. Activation 设计原则

1. **第一屏必须卖“风险判断”，不是卖“健康管理”**
2. **第一轮输入只围绕一个具体对象**
3. **追问数量上限要被严格约束**
4. **颜色后面必须立刻跟一个普通人能懂的原因**
5. **结果页必须帮助用户做出动作，而不是停在解释**
6. **保存不是收尾，而是下一次复查的预埋**

---

## 7. Activation 指标

### 核心指标

| 指标 | 定义 | 目标 |
|---|---|---|
| 查询开始率 | 进入首页后发起一次查询的用户占比 | 待上线后定义 |
| 首次查询完成率 | 用户完成输入并看到结果页 | >= 80% |
| 解释理解率 | 用户能复述“为什么不适合我” | >= 70% |
| 结果保存率 | 首次查询后保存到档案 | >= 40% |
| 强激活率 | 完成查询 + 理解原因 + 保存结果 | P0 核心追踪指标 |

### 辅助指标

- 追问平均题数
- 结果页停留时长
- 结果页后二跳率
- 保存前后的流失率

## 8. 事件埋点建议

- `view_home`
- `start_query`
- `submit_query_object`
- `answer_context_question`
- `view_result`
- `view_risk_reason`
- `expand_evidence`
- `select_next_action`
- `save_result_to_archive`
- `exit_without_save`

---

## 9. P0 的激活文案要求

### 首页必须回答

- 我能查什么
- 为什么不是所有补剂都一样
- 为什么它和“我的体质”有关

### 结果页必须回答

- 风险来自哪里
- 为什么和你有关
- 现在该怎么办

### 保存 CTA 必须回答

- 为什么现在要保存
- 保存后下次具体会省掉什么

---

## 10. 设计结论

**VitaMe P0 的激活，不是“用户看到了结果”，而是“用户第一次真正理解：补剂安全是个和自己强相关的判断问题，而 VitaMe 能把这个问题讲明白，并帮我把这次判断沉淀下来。”**
