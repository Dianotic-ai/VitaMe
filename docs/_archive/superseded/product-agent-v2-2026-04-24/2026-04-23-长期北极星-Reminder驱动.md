# VitaMe 长期北极星：Reminder 驱动的反馈闭环

> 日期：2026-04-23
> 状态：Draft（待团队确认）
> 关系：**不 replace 既有 P0 总纲**，是接在 P0 成功之后的 P1/P2 方向

---

## 1. 这份文档是什么、不是什么

**是：** 对 Kevin 在 2026-04-23 提出的产品 Pivot 的落地记录——北极星从"建档率"转向"服药依从率 + 反馈密度 + Memory 深度"，并用 Hermit Agent 实现自我迭代。

**不是：** 对 `P0-执行总纲.md` 的替换。P0 保持"补剂安全判断 wedge"不变，黑客松 Demo 照此走。本文件描述 **P0 成功之后**的 P1/P2 演化路径。

## 2. 核心判断的变化

### 2.1 北极星分层

| 阶段 | 北极星指标 | 成功信号 | 对应文档 |
|---|---|---|---|
| **P0**（黑客松） | Aha Moment 完成率 | 用户粘产品链接 → 看到 ✅/⚠️/❌ 判断 | P0 统一执行总纲 |
| **P1** | 日反馈率 | 提醒触发后用户应答率 > 40% | 本文件 |
| **P2** | Memory 深度 | 单用户 30 天反馈 > 60 条；Hermit 识别模式 ≥ 3 个 | 本文件 |
| **P3** | 大健康迁移 | 非补剂行为（睡眠 / 饮食 / 运动）接入同一套闭环 | 未来路线图 |

### 2.2 "档案"地位的变化

| | 旧设计（v2 SOP） | 新设计（本文件） |
|---|---|---|
| 档案的角色 | 入口 / 产品前提 | 副产品 / 随反馈自然沉淀 |
| 用户感知 | "请填写你的信息" | "帮你记住今天的感受" |
| 北极星 | 建档完成率 | 反馈密度 × 留存 |
| Intake Tier 的意义 | 4 档阶梯升级 | 最多 Tier 1（性别/年龄段/主要关切），其余字段由反馈长出来 |

**结论**：Tier 0/1 保留，Tier 2/3 被**反馈数据自然替代**。不再让用户"填写 5 分钟表单"。

## 3. 产品主循环（核心飞轮）

```
         ┌──────────────────────────────────────────────────┐
         │                                                  │
         ▼                                                  │
    加入一款补剂                                            │
    (拍瓶子/粘链接/说名字)                                   │
         │                                                  │
         ▼                                                  │
    自动识别属性                                            │
    (水溶/脂溶、时段、方式)                                  │
         │                                                  │
         ▼                                                  │
    设提醒 (用户一键确认)                                    │
         │                                                  │
         ▼                                                  │
    每次提醒 → 反馈仪式                                      │
    ("吃了吗？感受？")                                       │
         │                                                  │
         ▼                                                  │
    反馈沉淀 Memory                                         │
         │                                                  │
         ▼                                                  │
    Hermit Agent 周期扫描 → 归纳模式 → 优化提醒/建议        │
         │                                                  │
         └──────────────────────────────────────────────────┘
```

**新用户门槛 = 加一款补剂 + 确认提醒（< 30 秒）**。其他一切后续自生长。

## 4. Reminder 引擎的属性识别

| 属性 | 识别方式 | 规则示例 |
|---|---|---|
| **水溶 / 脂溶** | 成分查表（硬编码） | VC / B 族 = 水溶 → 可空腹；D3 / K2 / Omega-3 = 脂溶 → 随餐含油脂 |
| **时段** | 机制规则（硬编码） | Q10 早 / 午（抗疲劳）；镁 睡前（放松）；褪黑素 睡前 30min；铁剂 早饭前空腹 |
| **服用方式** | 禁忌查表 | 胃溃疡 + 铁剂 → 改饭后；钙 + 铁 → 不同时段；鱼油 → 含油脂餐 |
| **剂量拆分** | 每日多次 | B 族 → 早午两次避免夜尿；镁 → 睡前单次；Q10 → 早 + 午，避免睡前 |

MVP 阶段：前三类全走**硬编码查表**，LLM 只做"商品名 → 成分归一化"的语义匹配（如 `Now Foods Ultra Omega-3` → `EPA+DHA Omega-3, 肠溶`）。

## 5. Feedback Ritual（反馈仪式）

### 5.1 触发点

| 触发 | 文案示例 | 响应形式 |
|---|---|---|
| 吃后 + 2 小时 | "刚刚吃的 Q10 有什么感觉？精力 1-5 分？" | 1 个滑块 + 可跳过 |
| 未吃 + 提醒过去 30 分 | "今天没吃 Q10，心脏 / 精力有没有变化？是不是熬夜了？" | 2 选项多选 + 可跳过 |
| 连续 3 天未吃 | "最近没按时吃 Q10，要调整时间还是先停用？" | [调时间] / [暂停] / [继续] |
| 用户主动 | 顶部"今日记一笔" | 自由输入 + 标签 |

### 5.2 追问设计原则

1. **一次一问**，每次反馈不超过 2 个字段
2. **全部可跳过**，跳过不惩罚（不降权）
3. **选项 > 自由文本**，降低打字负担
4. **严重异常关键词硬编码拦截**：如出现"心悸""胸痛""剧烈不适""晕倒"等，立即触发"建议就医"页面，不走 LLM 推理
5. **避免诊断式措辞**：只用"你感觉到…吗？"不用"这可能是…导致的"

## 6. Memory 沉淀 + Hermit Agent

### 6.1 Memory 结构

每条 Memory = `时间戳 + 事件类型 + 关联实体 + 用户原话 + 情感/生理标签`

例：
```
2026-04-25 23:12 | missed_reminder | Q10 | "熬夜了，忘记吃" | {fatigue: 3, sleep: late}
2026-04-26 08:45 | feedback | D3+K2 | "精力比昨天好" | {energy: 4}
```

**Memory ≠ 传统数据库**。它是 agent-readable context——每次对话时按相关性召回 Top N 条喂给 LLM（Anthropic Memory API / MemGPT 模式）。端侧加密存储，远端只存派生哈希用于召回索引。

### 6.2 Hermit Agent 是什么

定义：一个"沉在用户数据底层"的 meta-agent，**用户不在时也在想你的 Agent**。

周期任务（每周 / 每月）：

1. **模式归纳**："你每次熬夜后第二天 100% 忘吃 Q10 → 要不要加睡前弹窗？"
2. **补剂建议**：基于连续反馈，提议新增 / 停用（例："最近 2 周你 5 次反馈胃不舒服都出现在 B 族服用后，要不要换成空腹前的活性 B？"）
3. **长期字段主动询问**："你的胃溃疡一个月前有反馈，现在情况如何？要不要调整 PepZin GI 剂量？"
4. **健康事件识别**："近 30 天 3 次反馈心悸 + 熬夜 → 建议去医院做动态心电图"

**Hermit ≠ 主 Agent**：
- 主 Agent = 响应用户提问，被动
- Hermit = 用户不主动时也在迭代，主动

这是从"工具"到"陪伴"的关键跃迁，也是长期留存和数据飞轮的核心。

## 7. 新用户门槛的降低（关键 Pivot）

### 7.1 错误的做法

"欢迎使用 VitaMe，请先完善您的健康档案（15 个字段）"  
→ 80% 流失。

### 7.2 正确的做法

| 步骤 | 用户动作 | 时长 | 隐式建档字段 |
|---|---|---|---|
| 1 | 粘一个补剂链接 → 看 Verify 结论 | 5 秒 | 无 |
| 2 | Agent: "要不要我帮你安排每天的服用时间？" | 点确认 = 同意 | `active_supplements: [X]` |
| 3 | 第一次提醒: "吃了吗？" | 点"是" | `adherence_rate: n/m` |
| 4 | 反馈仪式: "感受 1-5 分？" | 拖滑块 | `feedback_entries` |
| 5 | Agent 第 3 天主动问: "你吃 Q10 是早上还是下午更有精力？" | 点选 | `preference: timing.Q10 = morning` |

**每一次问题 = 一次 Aha + 一个字段**。档案 100% 自然生长，用户从未感知在"填表"。

### 7.3 保留的 Tier 0/1

| Tier | 触发时机 | 问题 |
|---|---|---|
| Tier 0 | 默认 | 无需 |
| Tier 1 | 第一次 Verify 时硬编码询问 1 个关键字段 | "这瓶是给谁用的？你自己 / 爸妈 / 另一半"（决定后续措辞与关切点） |
| Tier 2 / 3 | **不再作为独立流程** | 全部由反馈仪式自然补全 |

## 8. 与 P0 统一执行总纲的对齐

| P0 总纲条款 | 本方向是否冲突 | 如何处理 |
|---|---|---|
| P0 只做"补剂安全判断" | 不冲突 | 本文件全部是 P1+ 范围 |
| P0 不做长期调理 | 不冲突 | 长期调理在 P1+；P0 只埋"设提醒"钩子作为留存接口 |
| P0 不回答"今天身体不舒服怎么办" | **需警惕** | Feedback Ritual 会收集不适反馈，必须严格走"不诊断 + 关键词硬编码拦截 + 建议就医"分流，LLM 不得做病因推理 |
| P0 不做"最值得买的品牌" | 不冲突 | Reminder 引擎不涉及选品 |
| 硬编码禁忌规则不可覆盖 | 增强 | 服用方式（空腹 / 饭后 / 分时段）规则也加入硬编码库 v0 |
| 免责声明强制展示 | 增强 | 每次 Feedback Ritual / Hermit 建议都带免责声明 |

## 9. 长期愿景：从补剂到大健康

补剂是切口，不是终局。主循环 **（加事件 → 属性识别 → 提醒 → 反馈 → Memory → Hermit）** 是通用的，可以承接：

| P3+ 扩展方向 | 加事件的形式 | 属性识别 | 反馈仪式 |
|---|---|---|---|
| 处方药 | 拍药盒 / 医生方子 OCR | 更严格的时段 + 禁忌 | 副作用监测 |
| 睡眠 | 手环接入 / 手动记录 | 入睡 / 醒来时点 | 睡眠质量 |
| 饮食 | 拍餐 | 宏量营养 | 餐后感受 |
| 运动 | 手动 / 手环 | 强度 / 时段 | 精力 / 肌肉感 |
| 体检 | 报告上传（Tier 3 专用） | 多年趋势 | 长期建议 |

**北极星不变**：**通过长期反馈让 Agent 越来越懂你**。

## 10. 风险与边界

| 风险 | 表现 | 缓解 |
|---|---|---|
| **过度医疗化** | 反馈追问容易滑向"诊断" | 硬编码"Agent 不诊断，仅记录；严重关键词强制建议就医" |
| **隐私敏感度加倍** | 反馈数据 > 一次性病史敏感度 | Local-first、端侧加密、字段级开关 |
| **通知疲劳** | 每日多提醒 + 多反馈 = 打扰 | 默认 ≤ 3 次 / 日；连续 3 天无应答自动降频 |
| **Hermit 错误归因** | 模式归纳可能误关联 | Hermit 建议必须标注"基于反馈 X/Y 的观察，非医学结论"，且一键反馈"不准" |
| **定位漂移** | 从 Verify 钩子滑向"健康顾问" | 每个 Agent 输出仍锁死在 P0 总纲的 "4 个可回答问题"内，超出范围 = 建议就医 |

## Insights

- **档案降权 = 承认消费者产品的残酷真相**：没人愿意为一个未验证的产品填 5 分钟表。反过来，一旦用户每天和产品发生 3 次轻交互（提醒+反馈），30 天就自然有 90 条数据点——比任何表单都细。
- **Hermit Agent 是护城河，不是功能**：主 Agent 的功能任何竞品都能抄；但"沉在用户 30 天数据里、比用户自己还了解 Ta 的 Agent"不是一周能复制的——它需要时间 + 反馈密度。这是 VitaMe 值得长期做的原因。
- **从补剂切大健康的关键是"加事件→反馈"的通用性**：补剂只是第一种可以"每天加一次"的健康行为。真正的 moat 是这个 **事件 → 属性识别 → 提醒 → 反馈 → Memory** 的通用管道。补剂是证明管道有效的最轻量最低风险场景。
