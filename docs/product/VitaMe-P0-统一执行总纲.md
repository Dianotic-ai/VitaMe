# VitaMe — P0 统一执行总纲

> 日期：2026-04-20  
> 状态：Draft  
> 目的：把当前有效产品文档与 superpowers 工程包收口成一份唯一执行基线，避免 P0 再次滑回“泛健康顾问 / 导购 / 内容平台”。

---

## 1. 这份文档解决什么问题

当前仓库已经基本收敛到正确方向：

**补剂安全判断 → 安全翻译解释 → 保存 → 复查**

但执行层仍有 3 个容易跑偏的风险：

1. 把“翻译”做成价值主语，弱化“判断”
2. 把“给自己买”和“给父母买”同时当成首页主场景
3. 把 OCR、替代建议、长期档案系统做得比核心风险判断更重

所以这份总纲只做一件事：

**锁定 P0 到底做什么、先做什么、坚决不做什么。**

---

## 2. P0 的一句话定义

**VitaMe P0 是一个补剂安全判断工具：在用户下单或开吃之前，结合成分、病史和当前用药，告诉用户这款补剂是否值得警惕、为什么、以及下一步怎么更稳妥。**

### 方法，不是主语

- **“安全翻译”** 是交付方式
- **“风险判断”** 才是产品价值核心

---

## 3. P0 的唯一主问题

**这款补剂，对这个人当前是否存在已知风险或需要注意的点？**

P0 只回答 4 个问题：

1. 这是什么成分 / 形式
2. 结合当前人群上下文，它是否有已知风险
3. 风险来自哪里，证据强弱如何
4. 下一步应该避开什么，或换成什么方向更稳妥

P0 不回答：

- 我得了什么病
- 我应该怎么治疗
- 我应该长期怎么调理
- 我今天身体不舒服该怎么办
- 哪个品牌最值得买

---

## 4. 目标用户与首发切口

### Primary User（首发切口）

**给自己买补剂、并且对安全性有明确顾虑的决策型用户。**

判定标准：

- 已经准备购买 / 已经购买 / 正准备追加一种补剂
- 具有体质标签、病史标签、在用药物、特殊人群属性中的至少一项
- 不是来闲逛内容，而是来做一次具体决策

### Secondary User（可演示，不做首页主叙事）

**替父母做补剂决策的成年子女。**

保留原因：

- 风险感强，叙事打动人
- 适合演示“保存到档案 / 下次复查”

但不作为 P0 首页唯一主故事，否则会让：

- onboarding 变重
- 问题采集变复杂
- UI 文案两头摇摆

---

## 5. P0 的主链路

### 唯一主链路

**输入一款补剂 → 有限追问 2–4 题 → 风险判断 → 原因解释 → 风险规避方向 → 保存到档案 → 未来复查**

### 价值判断标准

P0 成立，不取决于用户聊得久，而取决于是否完成这 6 件事：

1. 用户成功输入一项补剂或成分
2. 系统只问最少必要问题
3. 结果能稳定落到红 / 黄 / 灰 / 绿
4. 用户能理解“为什么”
5. 用户知道下一步怎么做
6. 用户愿意把结果保存，形成下一次复查起点

---

## 6. P0 必须打穿的三件事

### Must Ship 1：风险判断可靠

- 高风险必须优先走确定性逻辑
- 未覆盖场景要明确落灰，不得假装安全
- 结果必须有依据类型和证据强弱

### Must Ship 2：解释可懂

- 用户能在 30–60 秒内看懂“为什么和我有关”
- LLM 只负责翻译，不负责改判定
- 高频 `reason_code` 必须有模板兜底

### Must Ship 3：保存与复查成立

- 第一次查询不是一次性结果页
- 结果可以保存给某个 Person
- 下次新增一项时可以带着既有 context 快速复查

---

## 7. P0 明确不做

1. 不做疾病诊断
2. 不做完整症状问诊
3. 不做长期症状追踪 / 打卡陪伴
4. 不做疗法、作息、饮食全套方案
5. 不做品牌导购、比价、CPS 变现
6. 不做开放式长对话健康顾问
7. 不做全量中成药与跨境药物覆盖
8. 不做“百科式补剂内容产品”

---

## 8. 产品与工程优先级排序

### 正确排序

1. **手动输入可用**
2. **风险判断可信**
3. **解释与规避建议可懂**
4. **保存与复查成立**
5. **OCR 作为入口增强**
6. **更复杂的数据源扩展**
7. **更完整的人群 / 国家 / 商品覆盖**

### 禁止颠倒

不得出现以下执行偏差：

- OCR 很炫，但判定不稳
- 页面很多，但没有一条链路全通
- 档案模型很复杂，但第一次查询体验很差
- 替代建议很多，但边界已滑向导购
- 讲了很多“AI Agent”，但没有稳定的输出契约

---

## 9. 北极星与成功标准

### 北极星指标

**Qualified Safety Decision（QSD）**
> 用户完成一次有效查询，并看到可解释的风险结果，且该结果被保存或触发明确下一步动作。

### P0 成功标准

| ID | 指标 | P0 目标 | 备注 |
|----|------|---------|------|
| SC-1 | 首次查询完成率 | >= 80% | 从输入到结果页 |
| SC-2 | 结果保存率 | >= 40% | 首次查询后 |
| SC-3 | 解释理解代理指标 | >= 60% | 点击“我知道为什么了”或查看证据/展开解释 |
| SC-4 | 二次回访率 | >= 25% | 30 天内再次查询或复查 |
| SC-5 | 主观信任度 | >= 4/5 | 访谈或结果页反馈 |

### 本阶段不追

- 伪 DAU
- 日活停留时长
- 内容消费深度
- 电商转化

## 10. 对外认知锚点

### 首页推荐文案

- Hero：**买补剂前，先查它适不适合你的体质**
- 副标题：**输入补剂、病史和当前用药，看懂这款补剂对你要不要警惕、为什么、该避开什么。**

### 不建议作为当前主锚点

- “拍一下，查安全”  
  原因：它会把团队注意力拉向 OCR，而不是判断可靠性。可作为次级功能文案，不应作为 P0 唯一认知主语。

---

## 11. 数据与责任边界

### 产品本质

- 我们做的是**风险识别与决策辅助**
- 不是安全背书
- 不是医疗建议
- 不是购物推荐

### 一条必须写死的边界

**“当前范围内未发现已知高风险” ≠ “绝对安全”。**

结果层必须明确区分：

- 已知高风险
- 已知注意事项
- 证据不足 / 暂无充分证据
- 当前规则和数据范围内未见明显风险

## 12. superpowers 风格的执行顺序（锁门版）

参考 superpowers 的“结构化计划 + 审批门 + 证据先于宣称”思路，P0 推荐按 5 个门推进，而不是边聊边改。

### Gate 1：总纲冻结

冻结以下问题：

- 一句话定位
- 唯一主场景
- P0 明确不做
- 北极星指标
- 首页主文案

### Gate 2：判断规则冻结

冻结以下产物：

- 用户上下文分类法
- 风险判定矩阵
- P0 数据白名单

### Gate 3：接口冻结

冻结以下产物：

- API Contract
- 前后端字段名
- 错误码
- 状态机

### Gate 4：测试冻结

冻结以下产物：

- 黄金样例
- 回归矩阵
- 合规禁词检查
- Demo 验收线

### Gate 5：范围冻结

到 Demo 前 5 天，若以下任一未打通，则立刻砍功能：

- 结果页不稳定
- 保存与复查不成立
- 合规护栏不稳定
- OCR 识别经常阻断流程
