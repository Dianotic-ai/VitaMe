# VitaMe — User Journey

## 1. Persona

首发 persona：正在给自己或父母买补剂的决策型用户。他不想研究营养学，也不想先填档案，只想快速知道“这瓶对我/家人是否需要警惕”。

长期 persona：已经开始服用补剂，并愿意用低摩擦反馈让 Agent 逐渐理解自己真实反应的人。

## 2. Journey Map

| Stage | Touchpoint | User Action | Emotion | Pain Point | Product Opportunity |
|---|---|---|---|---|---|
| Awareness | 小红书/电商/家人推荐 | 看到一款补剂 | 好奇 + 不安 | 营销话术太多 | 用“买前先查”建立尖锐入口 |
| Verify | 首页 / 查询页 | 输入自然语言或产品名 | 谨慎 | 不想填长表 | Verify-first，最多 2 轮追问 |
| Judgment | 结果页 | 看红黄灰绿和原因 | 放松或警觉 | 只看颜色不够 | 先结论，再人话解释，再证据 |
| Save | 结果页 CTA | 保存给自己/家人 | 犹豫 | 不知道保存价值 | 告诉用户“下次复查会少问” |
| Reminder | 保存后 | 确认服用提醒 | 控制感 | 怕被打扰 | 默认低频，可暂停/降频 |
| Feedback | 提醒后 | 点选是否服用和感受 | 轻负担 | 不想写日记 | 一次一问，全部可跳过 |
| Memory | 后台/可见摘要 | 查看被记住的反馈 | 信任或担心 | 数据去哪了 | 隐私 HUD + LLM 输入可见 |
| Hermit | 周报/月报/主动卡片 | 确认或否定 Agent 观察 | 被理解 | 怕被诊断 | 只输出观察，不做医疗归因 |

## 3. Critical Moments

### P0 Aha

用户第一次意识到：不是所有补剂都一样，成分形式、病史和用药会改变“能不能吃”。

### P1 Aha

用户发现：保存后不是进后台，而是能一键安排提醒，并在提醒后顺手记录感受。

### P2 Aha

用户看到 Hermit Agent 说出一个真实模式，例如“你最近 3 次胃不舒服都发生在晚间服用 B 族后”，并且能点“准 / 不准”。

## 4. Flow A — First Verify

1. 用户进入首页。
2. 输入：“我妈在吃华法林，能吃辅酶 Q10 吗？”
3. L0 解析成 ingredient + medication。
4. L2 输出风险。
5. L3 翻译成人话。
6. 结果页展示风险、原因、证据、免责声明、保存入口。

成功标准：用户能复述“为什么和我有关”。

## 5. Flow B — Save to Context

1. 用户点击保存。
2. 选择 self / mom / dad / other。
3. 系统保存本次查询摘要、上下文、风险和边界。
4. 页面说明“下次新增补剂时，会自动带入这些信息复查”。

成功标准：保存动作明显降低下一次输入成本。

## 6. Flow C — Reminder + Feedback

1. 保存后出现提醒建议。
2. 用户确认或调整时间。
3. 到点提醒：“今天吃了吗？”
4. 用户点“吃了 / 没吃 / 跳过”。
5. 若吃了，系统只问一个反馈字段。

成功标准：反馈像顺手确认，不像写健康日记。

## 7. Flow D — Memory + Hermit

1. 反馈进入 Memory。
2. Hermit 周期性扫描近期事件。
3. 系统生成可解释观察。
4. 用户确认、否定或忽略。
5. 被确认的观察影响后续提醒和复查。

成功标准：Agent 变得更懂用户，但用户始终可控。

## 8. Churn Triggers

- 第一屏看不懂和普通健康内容有什么不同。
- 一上来要求完整建档。
- 结果只有颜色，没有原因。
- 保存后没有实际用途。
- 提醒太频繁。
- Hermit 文案像诊断或因果归因。

## 9. Design Constraints

- 每一步只服务一个用户任务。
- 每个新增问题都必须带来即时价值。
- 任何长期能力都必须从一次 Verify 成功自然延伸。
- 隐私和合规边界必须进入主 UI，不藏在 FAQ。
