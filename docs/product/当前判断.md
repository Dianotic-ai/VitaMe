# VitaMe — 当前判断

> 本文件只保留“现在还应该影响决策”的判断。历史材料只作为证据，不作为日常阅读入口。

## 1. 当前一句话判断

VitaMe 应该做成一个**以补剂安全判断为入口、通过 Reminder + Feedback + Memory 自我进化的健康 Agent app**。

P0 的产品价值是“可信、可懂、可复查的安全判断”。黑客松表达需要补上 Agent shell：让 AI 展示如何理解输入、选择工具、调用确定性安全判断、解释结果，并把一次判断变成后续 Memory / Reminder 的起点。长期价值是“Agent 越来越懂用户的真实反应，并把这种理解转化为更轻的提醒、更准的复查、更少的健康决策焦虑”。

5 天上线范围锁定为 P0 主链 + Vercel AI SDK ToolLoopAgent shell + 北极星钩子展示。ToolLoopAgent 只做工具调度和行动建议，风险等级仍来自 L2。Mastra / LangGraph 后移为 P1/P2 或 durable graph 的后续评估项。

## 2. 分层判断

### P0 判断

- 先做窄入口：买/吃补剂前的安全排雷。
- 用户先看到一次明确判断，再决定是否保存。
- 风险结论来自 L2 规则和数据，不来自 LLM 自由判断。
- L3 只负责翻译结构化风险，不改判定。
- ToolLoopAgent 可作为 Demo shell，但只能调用 `parseIntentTool / runJudgmentTool / translateRiskTool / createActionPlanTool / createMemoryPreviewTool`。

### P1 判断

- 保存不是后台管理动作，而是 Reminder 和 Feedback 的入口。
- Reminder 的核心不是“提醒吃”，而是触发轻反馈。
- Feedback Ritual 必须一次一问、可跳过、低摩擦。

### P2 判断

- Memory 是护城河，不是传统表单档案。
- Hermit Agent 是周期性模式归纳器，不是主动诊断医生。
- 自我进化必须可解释、可撤销、可纠错。

## 3. 最新产品洞察

- **价值来自判断，不是内容。** 用户不是缺少补剂文章，而是缺少把个人上下文映射到风险结果的能力。
- **冷启动不能要求完整建档。** 先 Verify，再用“告诉我 X 可让判断更准”渐进收集信息。
- **档案应自然长出来。** 用户每天反馈 1 个小信号，比一次性填写 15 个字段更真实。
- **Hermit Agent 是长期壁垒。** 主 Agent 问答可以被复制，沉在用户 30 天反馈里的个体模式不能被一周复制。
- **隐私要产品化。** 用户需要看见哪些信息被使用、为什么被问、是否进入 LLM。
- **Demo 仍要打穿一条链路。** 长期愿景不能冲淡 P0 主链：输入 → 判断 → 解释 → 保存。

## 4. 当前优先级

1. P0 主链联调：`/api/intent → /api/judgment → /api/translation → result UI`。
2. P0 Agent shell：planned `/api/agent` 使用 ToolLoopAgent 调用白名单工具并展示 trace。
3. 文档事实源更新：PRD、Journey、Architecture、Data Flow、Test Matrix、Implementation Map。
4. P1 设计预埋：保存后提醒、反馈仪式、Memory schema、Hermit cycle 只写 planned。
5. 合规闭环：DemoBanner、disclaimer、禁词、医学审核状态、audit status。

## 5. 不再作为当前判断的旧方向

- “健康守护 Agent”作为泛定位过宽，只保留为历史背景。
- “先完整建档再出结果”会伤害冷启动，不采用。
- “OCR 是主场景”会把注意力拉离判断可靠性，P0 降级为入口增强。
- “导购/比价/渠道折扣”会污染信任边界，P0/P1 不采用。
- “开放式症状问诊”容易越过医疗边界，只保留严格受限的候选成分例外。

## 6. 冲突处理规则

- 产品文档与代码冲突时，不静默改代码。
- 现有 canonical 产品目标优先保留。
- 代码差异写入 `docs/engineering/specs/implementation-map.md`。
- 医学风险等级冲突必须进入医学审核，不由 Agent 自行裁决。
