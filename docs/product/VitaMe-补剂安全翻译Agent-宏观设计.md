# VitaMe — 补剂安全翻译 Agent 宏观设计

> 日期：2026-04-17
> 状态：Draft
> 目的：在页面级动线 spec 之前，先定义产品的总体结构和设计母型

---

## 1. 为什么先做宏观设计

在画页面之前，先要明确：

- 这到底是一个什么形态的产品
- 用户脑子里应该怎样理解它
- 顶层信息结构应该怎么分
- 哪些对象是产品的核心
- 哪些体验绝对不能被做成“泛健康顾问”

如果这些问题不先锁住，后面的页面 spec 很容易滑回：

- 聊天机器人
- 健康内容站
- 表单后台
- 家庭管理工具

而这些都不是当前 P0。

---

## 2. 当前产品的宏观定义

### 产品不是一个“聊天界面”

VitaMe 的本质不是对话本身，而是：

**以查询为入口的补剂安全决策系统。**

对话可以存在，但它只能服务于 3 件事：

1. 帮用户更快表达查询对象
2. 补齐影响结果的关键上下文
3. 解释“为什么”

### 产品不是一个“内容平台”

用户不是来学习一门完整营养学课程。

用户是来解决一个临近的具体决策：

**这款补剂对我 / 对我爸妈，到底合不合适。**

### 产品不是一个“后台管理系统”

档案存在的目的，不是管理本身。

它的意义是：

**让下一次查询更轻。**

---

## 3. P0 的总体产品结构

### P0 只有 3 个核心层

1. **查询层**
   - 用户输入一个具体补剂 / 成分 / 药物
   - 系统采集最少必要上下文

2. **判断层**
   - 系统输出红黄灰绿结果
   - 系统解释原因、证据和规避方向

3. **沉淀层**
   - 用户保存结果到个人或家人档案
   - 用户在下一次新增一项时发起快速复查

### 这 3 层对应的用户心智

| 产品层 | 用户问题 |
|---|---|
| 查询层 | 我该查什么 |
| 判断层 | 为什么有风险 / 为什么没风险 |
| 沉淀层 | 下次怎么更快 |

---

## 4. P0 的核心对象

后续页面和数据模型，都应该围绕这些对象展开：

| 对象 | 说明 | 为什么重要 |
|---|---|---|
| Query | 一次具体查询 | 产品的起点 |
| Product | 某个商品或补剂 | 用户最常输入的对象 |
| Ingredient | 标准化成分 | 支撑解释和映射 |
| Person | 查询对应的人 | 区分自己 / 父母 / 家庭成员 |
| Risk | 风险判断结果 | 产品价值核心 |
| Reason | 风险原因解释 | Aha 的来源 |
| Evidence | 来源和证据级别 | 建立信任 |
| Archive | 被保存的查询上下文 | 支撑复查和留存 |

---

## 5. P0 的顶层信息架构

### 顶层不是多 Tab 导航

P0 不应该一上来就做复杂导航或多栏目首页。

当前更适合的是：

### 结构 A：单主任务 + 一个次级资产入口

- **主任务入口**：开始排雷
- **次级入口**：查看档案 / 继续复查

### 页面族

1. **首页 / 查询页**
   - 卖点、输入、开始排雷

2. **关键问题页**
   - 用最少问题补齐上下文

3. **结果页**
   - 风险等级、原因、证据、下一步建议

4. **保存 / 档案页**
   - 结果沉淀、家人区分、复查入口

5. **快速复查页**
   - 基于已有档案新增一项，重新交叉检查

---

## 6. P0 的交互母型

### 母型 1：对象优先，而不是症状优先

默认从一个具体对象开始：

- 一个补剂
- 一个成分
- 一个药物

而不是从“我最近不舒服”开始。

### 母型 2：有限追问，而不是开放对话

交互应该像“聪明的安全问诊卡”，不是无限聊天窗口。

原则：

- 只问影响判断的上下文
- 问题数量可预期
- 追问结束后立即出结果

### 母型 3：结果页是核心，而不是输入页

用户真正要消费的是结果页。

所以设计中心应该放在：

- 颜色等级
- 风险原因
- 普通语言翻译
- 证据来源
- 下一步建议
- 保存动作

### 母型 4：保存是延续，不是收尾

保存动作的设计意义是：

**把一次结果，变成下一次更轻的查询。**

---

## 7. P0 的视觉与信息层级原则

### 第一层：结论

先让用户知道：

- 红 / 黄 / 灰 / 绿
- 能不能继续 / 要不要小心 / 是否数据不足

### 第二层：原因

紧跟着解释：

- 风险来自成分形式 / 病史 / 药物交互中的哪一项

### 第三层：证据

告诉用户：

- 这是数据库结论、文献结论，还是有限证据

### 第四层：动作

最后才是：

- 该停
- 该换
- 该复查
- 该保存

### 禁止的层级错误

- 先堆免责声明，压住结论
- 先堆证据，用户还没理解问题
- 先给长段教育内容，拖慢决策

---

## 8. P0 的信任系统

VitaMe 不是靠“会聊天”建立信任。

它的信任由 4 个部件组成：

1. **明确结论**
2. **可理解原因**
3. **证据透明**
4. **边界诚实**

所以在宏观设计上，任何页面都不能牺牲这 4 件事去换花哨体验。

---

## 9. P0 不应该长成什么样

### 不要长成 ChatGPT

- 大输入框
- 无限对话流
- 结果埋在长文本里

### 不要长成健康资讯站

- 大量科普卡片
- 复杂频道
- 内容优先于决策

### 不要长成家庭后台

- 一进来就是一堆档案列表
- 先管理，再判断

### 不要长成电商导购页

- 品牌、价格、购买入口压过安全判断

---

## 10. 页面级 spec 之前必须锁定的设计决策

在开始画页面前，以下 6 个宏观决策应视为已定：

1. **P0 是手机 H5 / 微信内 WebView 优先**
2. **P0 默认从单个具体对象开始查询**
3. **P0 只做有限追问，不做开放长对话**
4. **P0 以结果页为核心价值页**
5. **P0 保存动作服务于复查，不服务于管理**
6. **P0 的顶层结构是“查询 -> 判断 -> 沉淀”，不是多模块并列**

---

## 11. 一句话结论

**VitaMe P0 的宏观设计，不是做一个会聊天的健康助手，而是做一个移动端优先、结果页优先、复查闭环优先的补剂安全决策系统。**
