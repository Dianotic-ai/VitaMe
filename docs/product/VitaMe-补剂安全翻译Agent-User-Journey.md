# VitaMe — 补剂安全翻译 Agent User Journey

> 日期：2026-04-17
> 状态：Draft
> 基于：`VitaMe-补剂安全翻译Agent-定位.md`
> 目的：明确 P0 版本的关键用户路径、触发时机、情绪变化和产品机会点

---

## 1. 设计原则

### P0 的 Journey 原则

1. **先解决一个具体决策，再谈长期关系**
2. **每一步都要减少“我看不懂”的焦虑**
3. **每一次查询都要尽量沉淀为下一次更轻的决策**
4. **不要把用户带进泛健康顾问式的长链路**

---

## 2. 用户 Journey 总览

| Journey | 触发时机 | 用户目标 | P0 核心价值 |
|---------|---------|---------|-------------|
| J1 我准备买一款补剂 | 看到商品/课程/种草内容时 | 判断能不能买 | 快速排雷 + 安全翻译 |
| J2 我已经买了，怀疑买错了 | 下单后、开瓶前、服用后不安时 | 判断能不能继续吃 | 风险确认 + 原因解释 |
| J3 我想给父母买 | 给父母选购或更新补剂时 | 降低替父母决策的风险 | 家庭场景的交互检查 |
| J4 我下次再来复查 | 新增产品、换药、复购、体检更新时 | 降低重复输入成本 | 档案沉淀 + 快速复查 |

---

## 3. J1：我准备买一款补剂

### 触发

- 在电商看到一款补剂
- 在课程、视频、小红书看到推荐
- 被“高吸收”“活性形式”“适合熬夜人群”等话术吸引

### 用户心理

- 初始：有兴趣，但不确定
- 核心焦虑：我看不懂，这是不是营销话术
- 真正要答案：我这种体质能不能吃

### 用户路径

1. 用户进入首页，看到“买补剂前，先查它适不适合你的体质”
2. 输入产品名、成分名或药物名
3. 系统追问 2-4 个必要问题
4. 系统返回红黄灰绿结果
5. 系统解释“为什么”
6. 用户决定：放弃购买 / 继续购买 / 换一个更安全方向
7. 用户选择保存本次结果到档案

### 关键页面/交互

- 首页 Hero
- 查询输入框
- 关键风险信息采集
- 结果页
- 保存到档案 CTA

### 产品要交付的瞬间

- “原来不是补镁都一样”
- “原来我这个体质不适合这种形式”
- “原来商家卖点和真实适配性不是一回事”

### 成功定义

- 用户能说出“为什么不适合我”
- 用户能根据结果做出购买决策
- 用户愿意保存结果到档案

### 失败信号

- 用户只看到颜色，不理解原因
- 用户觉得产品只是另一个搜索结果页
- 用户查完后没有任何后续动作

## 4. J2：我已经买了，怀疑买错了

### 触发

- 已经下单，但开始犹豫
- 刚收到货，准备开吃前不放心
- 已经服用几天，突然看到风险信息

### 用户心理

- 初始：后悔和焦虑
- 核心问题：现在继续吃会不会有事
- 额外问题：风险到底有多大，需不需要立刻停

### 用户路径

1. 用户输入已购买产品
2. 系统追问病史、用药、特殊风险
3. 系统输出风险等级
4. 系统用普通语言解释风险来源
5. 系统给出规避方向
6. 用户保存结果，后续对同类产品复查

### 产品要交付的瞬间

- “不是我过度担心，是真的有风险”
- “不是所有镁都不行，是我买的这个形式不适合”
- “我现在知道该停、该换，还是可以继续”

### 成功定义

- 用户明确下一步动作
- 用户不会因为恐慌而离开
- 用户对 VitaMe 的信任提升

### 失败信号

- 结果太模糊，用户不知道该怎么办
- 文案像在做医疗判断，造成不适或不信任
- 用户没有保存结果，也没有完成复查沉淀

---

## 5. J3：我想给父母买

### 触发

- 给父母选鱼油、钙片、维生素 D、辅酶 Q10 等常见补剂
- 父母现有处方药较多，不敢凭感觉买
- 父母听朋友推荐买了东西，子女想二次确认

### 用户心理

- 初始：责任感强
- 核心焦虑：买错不是浪费钱，而是可能出事
- 情绪特征：恐惧、内疚、谨慎

### 用户路径

1. 用户输入父母要买或正在吃的补剂
2. 系统追问父母病史和现有处方药
3. 系统输出风险等级和原因
4. 用户保存到“妈妈”或“爸爸”档案
5. 下次新增一项时，直接复查全部组合

### 产品要交付的瞬间

- “我终于不是靠猜给父母买东西”
- “我能把这件事做得更稳，不用只相信带货视频”

### 成功定义

- 用户愿意创建家人档案
- 用户愿意在下一次新增补剂时回来复查
- 用户愿意把结果转发给家庭中的另一个决策者

### 失败信号

- 录入成本太高，用户中途放弃
- 用户无法区分“补剂风险”和“必须就医”的边界
- 没有形成家庭场景的沉淀价值

---

## 6. J4：我下次再来复查

### 触发

- 新增一款补剂
- 换了药
- 打算复购
- 体检结果有新变化

### 用户心理

- 初始：这次应该更快
- 核心期待：不要再让我重输一次

### 用户路径

1. 用户进入已有档案
2. 点击“新增一项”
3. 输入新产品或新药物
4. 系统自动和既有档案做交叉检查
5. 输出新增风险或风险变化

### 关键价值

- 从“单次查询工具”升级成“可复查系统”
- 输入成本显著下降
- 每次新增都能成为自然回访理由

### 成功定义

- 用户认为第二次明显比第一次更省力
- 用户形成“先查再买”的习惯

### 失败信号

- 每次回访都像首次使用
- 档案没有真实减少输入成本
- 用户不明白为什么要保存结果

---

## 7. 情绪曲线

### J1 / J2 / J3 共通情绪变化

1. **困惑或恐惧**
   - 我看不懂，也不敢乱吃
2. **谨慎输入**
   - 我愿意回答几个关键问题，但不想被盘问太久
3. **瞬时理解**
   - 原来问题出在成分形式 / 病史 / 药物交互
4. **决策确认**
   - 我知道该停、该换、还是继续
5. **信任沉淀**
   - 这次帮我看懂了，下次我还会先来查

### 产品设计要求

- 不要让用户在困惑阶段停留太久
- 每个 Journey 都必须尽快产生“我明白了”的瞬间
- 结果页要从“颜色”走到“理解”

---

## 8. P0 页面流

### Flow A：首次查询

首页 → 输入产品/成分/药物 → 关键问题采集 → 结果页 → 保存到档案

### Flow B：家人场景

首页 → 输入补剂 → 选择“给谁查” → 关键问题采集 → 结果页 → 保存到家人档案

### Flow C：复查

档案页 → 新增一项 → 自动交叉检查 → 风险变化结果

---

## 9. Journey 对功能的约束

### 必须有

- 快速输入
- 有限追问
- 清晰结果等级
- 原因解释
- 保存到档案

### 可以后做

- 复杂症状历史
- 长期打卡
- 多人协作流
- 体检 OCR
- 详细替代方案库

### 明确不能混进 P0

- 泛健康问诊
- 长链路疗法推荐
- 每日陪伴式健康管理

---

## 10. 设计结论

P0 的 User Journey 不是“陪用户每天变健康”，而是：

**在用户每次即将做补剂决策时，用最短路径把复杂知识翻译成安全判断，并把这次判断沉淀成下一次更轻的决策。**
