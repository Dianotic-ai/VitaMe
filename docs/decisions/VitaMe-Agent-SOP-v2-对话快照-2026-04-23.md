# VitaMe Agent SOP v2 — 对话快照（2026-04-23）

> 本文件是 2026-04-23 上午对话中产出的 v2 SOP 原样留档。
> 同日 Kevin 提出"Reminder 驱动"pivot，导致"建档"从北极星降级为副产品。
> v2 里的 **"Intake 渐进 + 隐私 HUD + Verify-first 多入口"** 三块设计依然有效，并入了 pivot 后的长期北极星文档。
> 与 `VitaMe-P0-统一执行总纲.md` 冲突时，以总纲为准（P0 锁死在"补剂安全判断 wedge"）。

## 起因

基础材料：`docs/research/gemini-health-consultation.md`（1236 行）——某 40 岁男性在 Gemini Pro 进行的 14 轮补剂咨询。问题是：**如何把这 14 轮压缩成一个"缩短路径 + 智能体验 + 无隐私暴露"的 Agent SOP**。

后续 Kevin 补充两个约束：
1. 降低新用户建档门槛
2. 正视隐私敏感用户的顾虑

---

## 一、从 14 轮对话提炼的骨架

| 阶段 | 轮次 | 用户在做什么 | Gemini 漏在哪 |
|---|---|---|---|
| A. 建档 | 1, 6, 7, 10 | 挤牙膏式提供病史 | 无持久化 memory，信息分 4 次才补全 |
| B. 成分方案 | 1, 8, 9, 11 | "吃什么、多少、分阶段" | 每次从头推导，不增量 |
| C. 品牌选择 | 2, 3, 4, 5, 12 | "哪个牌子" | 无品牌库，训练数据陈旧 |
| D. 购物车校验 | 13, 14a-e | "我选的对吗" | 无批量解析机制 |
| E. 外部信息消化 | 4, 12 | "YouTube/ConsumerLab 说的呢" | 被动吸收 |

真实心智 = **建档 → 方案 → 选品 → 排雷**。

## 二、Gemini 四个根本缺陷

| 缺陷 | 症状 | Agent 要做的事 |
|---|---|---|
| 无持久化 memory | A 阶段重来 4 次 | 一次建档，增量更新 |
| 无结构化 intake | 自由文本 → 漏信息 → AI 反问 | 表单/引导式问卷 |
| **无冷启动阶梯** | 一上来就要完整画像才能给方案，80% 新用户会流失 | Intake 分档，Verify 可先跑 |
| **无隐私可视性** | 用户把病史、医院、数字贴进公共对话框，完全不知道数据到哪 | 隐私 HUD 常驻 UI |

后两项是产品化关键——Gemini 是 AI 工具，VitaMe 是消费产品。

## 三、SOP v2 总览

```
┌── 🔒 隐私 & 信任 HUD（常驻 footer）──────────────────────┐
│  Local-only  |  🔍 查看 LLM 看到什么  |  🗑 一键清档    │
└──────────────────────────────────────────────────────────┘

多入口（任选其一开始，不强制注册）：
 ┌─────────┬─────────────┬──────────┬──────────┐
 │ 问一句  │ 粘贴产品链接 │ 上传报告 │ 新手引导 │
 └────┬────┴──────┬──────┴────┬─────┴─────┬────┘
      ↓           ↓           ↓           ↓
   ②Compose   ④Verify     ①Intake     ①Intake
   (Tier 0)   (Tier 0)    (Tier 3)    (Tier 1)
                  │
                  │ 想要更准？
                  ▼
           ①Intake 渐进升档（Tier 0→1→2→3）
                  │
                  ▼
           ②Compose → ③Source → ④Verify（精准版）
                  │
                  ▼
       📁 Memory v1（用户明示同意才建）
                  │
                  ▼
       后续使用：新事件 → 局部重算，从不重做
```

三个核心设计：**Verify-first 多入口**、**Intake 四档渐进**、**隐私 HUD 可视**。

## 四、核心设计①：Verify-first 多入口

**原则**：5 秒内必须见到价值。

- 产品定位"25-40 岁成年子女给父母选购"——主场景就是"拿到一瓶东西想知道能不能吃"，所以 ④Verify 必须能独立先跑
- 入口不是"请填写你的健康信息"，而是"粘个链接 / 拍个瓶子 / 问一句"
- Tier 0 的输出永远标注："基于通用人群估计——告诉我 X 可让判断个性化 ✅/❌"，用钩子牵引升档

## 五、核心设计②：Intake 四档 + 渐进晋级

| Tier | 时长 | 要求 | 解锁能力 | 适合 |
|---|---|---|---|---|
| **0 匿名** | 0 秒 | 无 | ④Verify 通用人群打标 | 新用户、强隐私敏感 |
| **1 轻档** | 30 秒 | 性别/年龄段/3 项病史大类多选 | ②Compose 粗方案 + ④Verify 定向排雷 | 体验用户、帮爸妈看 |
| **2 标准** | 2-3 分钟 | 结构化病史 + 用药 + 症状 | 完整 ②③④ | 真正想拿方案的人 |
| **3 深度** | 5+ 分钟 | 报告 OCR / ICD 编码 / 多年趋势 | Memory 持久化 + 医生版摘要 | 长期管理 |

**渐进晋级**：不是进门就问你要升到几档，而是当下一步精度需要信息时 Agent 主动说：

> "要让这条建议更准，我需要知道你是否有结石病史。
>  🔒 这个字段只存在你的设备上，不发给服务器。[告诉 Agent] [跳过]"

**每次问一项、带即时价值兑现**——这比 5 分钟表单更智能的根本原因：用户感知到"答了这个，回答立刻变准"。

## 六、核心设计③：隐私 HUD 四件套（可见 = 可信）

| 控件 | 用户看到什么 | 背后做什么 |
|---|---|---|
| 🔒 **模式徽章** | 常驻显示当前模式：`Local` / `Pseudonymous` / `Cloud-Memory` | 模式决定数据路径，用户可切 |
| 🔍 **"LLM 看到的"** | 点开显示实际发给 LLM 的脱敏 token：`45M + N20.0 + K25.9` | 让用户眼见"姓名/医院/具体数字没上云" |
| 🎚 **字段级开关** | 每条病史旁有 `本地 / 上行` 开关 | 不上行的字段只喂本地硬规则，不进 Prompt |
| 🗑 **一键清档** | 顶部"清空 + 导出密钥" | 本地加密 key 可导出；清除后服务端密文失效 |

**关键设计**：隐私不是藏在设置里的后端实现，而是 UI 上能操作的开关。Gemini 最大的隐私问题不是"它存了什么"，是**"用户完全没感知自己交出了什么"**。让"看得见"变成产品差异点。

## 七、每步的 I/O（适配冷启动 + 隐私）

| 步 | 输入 | 处理 | 输出 |
|---|---|---|---|
| ④Verify（可先跑） | 一条 SKU / 商品名 / 链接 | 链接解析 + 通用人群风险扫描 + 反问 1 个关键差异点 | 风险标签 + "告诉我 X 可个性化" 钩子 |
| ①Intake（渐进） | 按档位逐项应答 / 报告上传 | 端侧 OCR + ICD 编码 + 本地加密；字段开关决定是否上行 | 档案快照（Tier 0/1/2/3） |
| ②Compose | 任意档位档案 | 硬编码禁忌过滤 → LLM 分阶段方案 → 剂量校准 | 成分表（档位越高越细）+ 免责声明 |
| ③Source | 成分表 + 偏好（国产/进口/预算） | 品牌库（NSF/USP/ConsumerLab）+ 实时价格 + 资质校验 | SKU 清单 + 渠道说明 |

档位约束：Tier 0 的输出 = 人群通用；Tier 3 = 个性化分阶段。全档位都带免责声明 + 隐私 HUD。

## 八、四种用户路径对照

| 画像 | 典型场景 | 走的路径 | 体感 |
|---|---|---|---|
| **帮爸妈看看** | 拿到一瓶补剂 | Tier 0 粘链接 → ④Verify → 被勾住后升 Tier 1 | 5 秒见价值、30 秒升档 |
| **隐私敏感自己用** | 有病史但不愿留数据 | Tier 0/1 + Local-only + 上行开关全关 | 全程不上云仍有用 |
| **给自己做方案** | 像那位 40 岁总监 | Tier 2 一次做完 + Memory | 比 Gemini 14 轮省 12 轮 |
| **长期管理** | 每 2 周回来一次 | Tier 3 + 增量更新 | 再次使用 <30 秒 |

四条路径共用同一套 SOP v2——区别只在用户走到哪一档就停。

## 九、与 VitaMe 既有决策的对齐

| 已落地的决策 | 在 SOP v2 里怎么兑现 |
|---|---|
| "禁忌规则硬编码，不可被 Prompt 覆盖" | ②Compose 硬过滤层；Tier 0 就是它的"人群子集"模式 |
| "免责声明每次 AI 输出强制展示" | 并入隐私 HUD footer：`非医疗建议 \| 🔒 Local \| 🔍 LLM 输入` 一体显示 |
| 价值链路"安全检测→成分解析→替代推荐+比价" | Tier 0-④ → ② → ③ 恰好对应 |
| MVP 不做 OCR、比价爬虫 | MVP 只实现 Tier 0/1 + 静态品牌库 + 人工维护禁忌规则；Tier 2/3 + OCR + 实时价格放 P1 |
| 定位"成年子女给父母选购" | **Tier 0 "帮爸妈看看"本身就是主入口**，反向确认 Verify-first 是正确的交互骨架 |

## 十、v1 → v2 的关键改动

1. 入口从 ①Intake 前置改为 ④Verify 可先跑（解决冷启动流失）
2. ①Intake 从"一次 5 分钟"改为"四档渐进、按需升档"（解决新用户怕填）
3. 隐私从"后端实现"升格为"UI 常驻 HUD"（解决隐私顾虑）
4. Memory 从"默认建档"改为"用户明示同意才建"（信任边界）
5. 每轮 LLM 输出加"钩子句式"：`告诉我 X 可让这条变准` → 把升档做成产品体验而非额外步骤

## Insights（原样保留）

- **"产品即建档"是冷启动的唯一解**：Gemini 是"先问病史再给建议"，VitaMe 应该是"先给建议再顺手收集一项信息"。每一次交互都在建档，但用户感知是在"拿到价值"——这是所有成功消费型 AI 产品（Cal AI、Finch、Flo）的共同模式。
- **隐私的产品化关键不是"更安全"，是"更可见"**：同样是本地加密，用户看得到开关 vs 藏在 FAQ 里，信任感差一个数量级。结合健康类产品的特殊性（用户天然警惕），把隐私 HUD 做成主 UI 元素，反而会变成"别人抄不走的差异点"。
- **Tier 0 能跑"通用人群模式"的前提**：50 条硬编码禁忌规则必须能在"没有具体档案"时也触发（按商品成分 × 常见高风险人群交叉打标）。这反过来定义了规则库 v0 的最低完备度——不只是给个性化用，也是 Tier 0 的引擎。
